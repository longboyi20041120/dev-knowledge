# CLAUDE.md — 编程知识库

这是你的私人编程知识库。你就是维护者——读取、整理、写入、索引，全部由你自动完成。

---

## Vault 结构

```
编程知识库/
├── raw/              ← 用户扔原材料的地方，你只读不删
│   ├── 网页剪藏/     ← 网页文章、博客、文档
│   ├── 报错日志/     ← 复制粘贴的错误信息、堆栈
│   └── 临时想法/     ← 一句话灵感，待你展开
├── wiki/             ← 你维护的结构化知识（核心产出）
│   ├── 工作学习/     ← 能写进简历的硬技能，面试用
│   ├── 项目/         ← 每个项目的完整知识
│   ├── 前沿信息/     ← 行业趋势、新技术动态
│   ├── 灵感/         ← 展开后的想法和创意
│   ├── 踩坑记录/     ← Bug 修复经验
│   ├── 索引.md       ← 总目录，每次写入都要更新
│   └── 最近更新.md   ← 变更日志
└── 模板/             ← 笔记格式模板，生成时参考
```

## 四大分类说明

| 分类 | 目录 | 用途标签 | 定位 |
|------|------|----------|------|

## 你的核心任务

### 1. 被动触发：用户问问题时

当用户问技术问题、查 bug、问项目细节时：
1. **先查 `wiki/索引.md`** 看有没有相关笔记
2. **再查对应目录**（学习→`wiki/工作学习/`，项目→`wiki/项目/`，趋势→`wiki/前沿信息/`，bug→`wiki/踩坑记录/`）
3. 找到答案直接用，找不到就诚实说"知识库里还没有"
4. 用户解决后，建议是否要写进知识库

### 2. 主动处理：检测到 raw/ 里有新文件

**每次打开项目时**，第一步检查 `raw/网页剪藏/` 有无未处理的日报（文件名 `*-日报.md`，开头没有 `<!-- processed` 标记的）。

处理日报的流程：
1. 先过滤：只保留与**知识体系**相关的内容。不相关的直接删掉（在日报末尾标注已过滤条目）
2. 筛选标准：能写进简历的技术、面试常考点、有实际代码示例的知识

**知识体系过滤器**（只有命中以下领域的才保留）：

| 领域 | 关键词 |
|------|--------|
| AI/机器学习 | LLM, Agent, Transformer, 深度学习, 神经网络, NLP, CV, 强化学习, 模型训练/推理/部署, Prompt |
| 编程语言 | Python, Java, C/C++, R, JavaScript/TypeScript, Go, Rust, 编译器/解释器, 类型系统 |
| 数据库 | SQL, MySQL, PostgreSQL, NoSQL, HBase, 索引优化, 查询优化 |
| 大数据 | Hadoop, Spark, Hive, Flink, Kafka, 数据仓库, 数据管道, ETL |
| 数据分析/可视化 | pandas, 数据可视化, BI, ECharts, matplotlib, 统计分析 |
| 操作系统/Linux | 进程, 内存管理, 文件系统, 内核, Shell, 系统调用 |
| 软件工程/架构 | 设计模式, 微服务, MVC, SOLID, CI/CD, DevOps, 测试, Git |
| 计算机基础 | 数据结构, 算法, 网络协议(HTTP/TCP), 计算机组成, 编码 |
| 数学/统计 | 概率论, 线性代数, 微积分, 离散数学, 假设检验, MLE |
| 容器/云原生 | Docker, Kubernetes, 容器化, 编排 |
| Web开发 | HTTP, REST, 前后端, Servlet, Spring, API 设计 |

**必须过滤掉的内容**：
- 纯个人博客（职业迷茫、"我的旅程"、心态分享）
- 非技术新闻（隐私政策社论、讣告、考古、社会科学）
- 纯硬件/物理/化学/生物（与CS无关的基础科学）
- 非编程工具的测评/推荐（键盘、显示器、椅子）
- 游戏（非教育/编程类）
- 元内容（"本周精选"、"Top 10 文章"）
3. **根据内容性质决定放入哪个分类**：
   - 实用技术、面试知识 → `wiki/工作学习/`
   - 行业趋势、新工具动态 → `wiki/前沿信息/`
   - 灵感创意 → `wiki/灵感/`
4. 每篇日报最多产出 2-3 篇笔记，**宁缺毋滥**
5. 日报顶部标记 `<!-- processed: YYYY-MM-DD -->`

其他 raw/ 处理规则：
- **网页剪藏**其他文件 → 同上
- **报错日志**有新文件 → 分析根因 + 解决方案 → 写入 `wiki/踩坑记录/`
- **临时想法**有新文件 → 展开论证（可行性、优劣、下一步） → 写入 `wiki/灵感/`
- 处理完毕后文件保留在 raw/（不删）
- **日报→周报→月报聚合**：规则见 [[日周月报规则]]。按月内固定分周（1-7 / 8-14 / 15-21 / 22-28 / 29-月末，不跨月），一周的日报齐了就建周报，当月周报齐了就建月报，**没满不建**；周报/月报头部用 `[[]]` 链接下级报表保证图谱连通。

### 3. 编码后：建议记录

改完代码后主动建议：
- 修了个值得记录的 bug？→ "要不要把这个 bug 记进知识库？"
- 做了个重要的架构决策？→ "这个决策值得写入 `wiki/项目/xxx/架构决策.md`"
- 学到了新的技术用法？→ "这个可以写入 `wiki/工作学习/`"

## 笔记格式规范

每篇笔记必须包含 YAML frontmatter：

```yaml
---
tags:
  - 标签1
  - 标签2
created: YYYY-MM-DD
updated: YYYY-MM-DD
status: draft | reviewed | archived
---
```

### 标签体系

| 维度 | 标签 | 说明 |
|------|------|------|
| 用途（必选） | `#用途/工作学习` `#用途/项目` `#用途/前沿` `#用途/灵感` | 决定笔记放在哪个目录 |
| 类型（必选） | `#类型/技术` `#类型/项目` `#类型/bug` `#类型/灵感` | 笔记的本质属性 |
| 技术栈（可选） | `#技术/ai` `#技术/python` `#技术/java` `#技术/spring-boot` `#技术/threejs` `#技术/mysql` `#技术/算法` `#技术/数据分析` `#技术/大数据` | 涉及的具体技术 |
| 状态（必选） | `#状态/草稿` `#状态/已验证` `#状态/已过时` | 成熟度 |
| 项目（可选） | `#项目/map_web` | 关联的具体项目 |

### 双向链接

- 用 `[[笔记名]]` 链接到其他笔记
- 每篇笔记至少有一个入链和一个出链
- 人在 `wiki/索引.md` 中能找到每一篇笔记

## 索引维护

**`wiki/索引.md` 是你最重要的文件**。每次在 wiki/ 下新增、删除、重命名笔记，都必须更新索引。

索引格式：
```markdown
## 工作学习
- [[工作学习/AI底层原理|AI 底层原理]] — 监督/无监督学习、深度学习、Transformer

## 项目
- [[项目/map_web/项目概览|map_web 项目概览]] — 校园导航系统

## 前沿信息
- [[前沿信息/Agent-Skill生态|Agent Skill 生态]] — 2026 Skill 爆发趋势

## 灵感
- [[灵感/xxx|xxx]] — 简述想法

## 踩坑记录
- [[踩坑记录/xxx|xxx]] — 简述问题和解法
```

**`wiki/最近更新.md`** 记录最近 10 条变更，格式：
```markdown
| 日期 | 笔记 | 操作 |
|------|------|------|
| 2026-06-22 | [[工作学习/AI底层原理]] | 新建 |
```

## 规则

- 写的笔记**人是第一读者**——清晰、有结构、不用代码块堆砌
- 不确定该不该写的时候，**问用户**
- 不删除 raw/ 里的任何文件
- 笔记状态从 `draft` 开始，用户确认后改为 `reviewed`
- 假设用户会用 Obsidian 的 Graph View 浏览关系图——链接密度要够
