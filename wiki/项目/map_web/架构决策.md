# 架构决策记录 — map_web

## ADR-1: 为什么前端是单文件 1.html

**时间**：项目早期
**决策**：所有前端代码放在一个 HTML 文件里（~3500 行），不拆分为 JS 模块。

**理由**：
- 校园导航是单一功能页面，不需要路由和模块化
- 减少网络请求，加载快
- 开发阶段改动方便，不用配置构建工具
- Three.js 从 CDN 以 ES Module 方式加载

**代价**：
- 文件长，可维护性随功能增长而下降
- 多人协作困难
- 如果功能继续增长，应该考虑拆分

## ADR-2: 为什么用模型名称而不是 ID 匹配

**时间**：3D 模型导入时
**决策**：前端用建筑名称（字符串）与 GLB 模型节点匹配，而非数字 ID。

**理由**：
- SketchUp 导出的 GLB 节点自带名称
- 比数字 ID 更直观，方便调试
- 不需要维护额外的映射表

**代价**：
- 名称变更时需要前端和后端同步
- 中英文混合名称容易出现编码问题

## ADR-3: AI 服务为什么用反向代理而不是直接调用

**时间**：AI 功能开发时
**决策**：前端统一访问 Java :8080，Java 通过 `AiProxyController` 反向代理转发到 Python :8000。

**理由**：
- 统一端口，前端不需要知道 Python 服务的存在
- 未来可以在代理层加认证、限流、日志
- 避免浏览器跨域问题

**代价**：
- 多一层转发，增加少量延迟
- Java 服务成为单点瓶颈

## ADR-4: AI 响应用 NAV 标记而不是直接返回命令

**时间**：AI 导航功能设计时
**决策**：AI 返回文本中的 `[NAV:fromId:toId]` 标记，前端解析后自动触发导航。

**理由**：
- 用户体验好：AI 说"从一教到食堂这样走"，地图同步画出路线
- 解耦：AI 不需要知道前端如何导航，只需要返回标准化标记
- 可扩展：未来可以加更多标记类型

**代价**：
- 前端需要维护标记解析逻辑
- 标记格式变更需要前后端同步

## ADR-5: 响应式布局用 CSS 变量 + clamp 而不是媒体查询堆叠

**时间**：2026-07-12
**决策**：面板宽/间距全部改为 CSS 自定义属性 + `clamp()` 平滑缩放，仅保留 3 个断点做模式切换（浮动卡片/折叠面板）。

**理由**：
- 纯媒体查询方案在 960px 半屏等常见场景下出现面板重叠
- `clamp()` 让面板随窗口平滑缩放，不跳变
- 面板位置通过 CSS 变量互相计算（如 `--info-left = --gap + --left-w + --gap`），不再硬编码 px
- 建筑信息面板被搜索栏碰撞的问题通过 "≤1300px 居中浮动卡片" 解决

**代价**：
- `clamp()` 嵌套 `calc()` 在某些旧浏览器不兼容（支持率 ~96%）
- 新增折叠按钮需要用户适应

## ADR-6: 正交相机水平视野固定模型大小

**时间**：2026-07-12
**决策**：frustumSize 从"垂直固定"改为"水平固定"，resize 时只调整垂直视野。

**理由**：
- 原来垂直固定 2000，窗口拉宽时建筑变窄，拉高时建筑变扁
- 改为水平固定 2000 后建筑大小恒定，更符合地图应用直觉
- resize 时显式重设 `pixelRatio` 防止高 DPI 屏幕分辨率下降

**代价**：
- `handleResize` 需要增同步更新 `renderer.setPixelRatio()` + `renderer.setSize()`
