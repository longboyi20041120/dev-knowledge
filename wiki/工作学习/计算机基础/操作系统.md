# 操作系统

> 参考教材：《计算机操作系统（第3版）》。操作系统面试八股文的半壁江山。这篇文章把进程、内存、文件系统、调度讲透。

## 一、操作系统的四大职能

操作系统的本质是**硬件资源和用户程序之间的管家**。

```
┌────────────────────────────────┐
│        用户/应用程序             │
├────────────────────────────────┤
│          操作系统               │
│  ┌──────┐ ┌──────┐ ┌────────┐ │
│  │进程管理│ │内存管理│ │文件管理 │ │
│  │(CPU) │ │(内存)│ │(磁盘)  │ │
│  └──────┘ └──────┘ └────────┘ │
│  ┌──────────────────────────┐ │
│  │       设备管理 (I/O)       │ │
│  └──────────────────────────┘ │
├────────────────────────────────┤
│    硬件 (CPU/内存/磁盘/外设)      │
└────────────────────────────────┘
```

**一句话总结四大职能**：
- **进程管理**：决定谁用 CPU、用多久
- **内存管理**：给程序分内存、用完回收
- **文件管理**：数据怎么存到磁盘、怎么找
- **设备管理**：键盘鼠标打印机等外设的统一接口

## 二、进程与线程

这是操作系统面试的最高频考点，没有之一。

### 进程 vs 线程

| 维度 | 进程 | 线程 |
|------|------|------|
| 定义 | 程序的执行实例 | 进程中执行的单元 |
| 资源 | 独立的内存空间、文件描述符 | 共享进程的内存空间 |
| 开销 | 创建/切换开销大 | 创建/切换开销小 |
| 通信 | IPC（管道、消息队列、共享内存） | 直接读写共享变量 |
| 独立性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能拖垮整个进程 |
| 类比 | 一个工厂 | 工厂里的工人 |

```c
// Linux 下的进程创建（概念示意）
pid_t pid = fork();
if (pid == 0) {
    // 子进程：pid == 0
    printf("我是子进程\n");
} else if (pid > 0) {
    // 父进程：pid > 0，返回的是子进程的 ID
    printf("我是父进程，子进程 ID = %d\n", pid);
}
```

```python
# Python 中的多线程 vs 多进程
import threading
import multiprocessing
import time

def worker(name):
    print(f"Worker {name} 开始")
    time.sleep(1)
    print(f"Worker {name} 结束")

# 多线程——适合 I/O 密集型（网络请求、文件读写）
t1 = threading.Thread(target=worker, args=("A",))
t2 = threading.Thread(target=worker, args=("B",))
t1.start(); t2.start()
t1.join(); t2.join()

# 多进程——适合 CPU 密集型（计算），绕过 GIL
p1 = multiprocessing.Process(target=worker, args=("X",))
p2 = multiprocessing.Process(target=worker, args=("Y",))
p1.start(); p2.start()
p1.join(); p2.join()
```

**Python GIL 问题**：CPython 有一个全局解释器锁（GIL），同一时刻只有一个线程能执行 Python 字节码。所以 Python 多线程对 CPU 密集型任务没用，但对 I/O 密集型有用（因为 I/O 等待时会释放 GIL）。

**面试话术**："进程是资源分配的基本单位，线程是CPU调度的基本单位。进程间隔离好但通信复杂，线程共享内存通信方便但一个挂了可能影响全局。需要 CPU 密集用多进程，I/O 密集用多线程。"

### 进程状态转换

```
        ┌──────────────────────┐
        │        创建           │
        └──────────┬───────────┘
                   │ 分配资源
                   ▼
        ┌──────────────────────┐
   ┌───│        就绪态          │◄─────────┐
   │   │   (Ready: 等CPU)      │           │
   │   └──────────┬───────────┘           │
   │         调度 │           时间片用完    │
   │              ▼                       │
   │   ┌──────────────────────┐           │
   │   │        运行态          │──────────┘
   │   │   (Running: 在CPU上)  │
   │   └──────────┬───────────┘
   │       等待事件│
   │              ▼
   │   ┌──────────────────────┐
   │   │        阻塞态          │
   └───│  (Blocked: 等I/O)    │
       └──────────────────────┘
                  │ 事件完成
                  ▼
       ┌──────────────────────┐
       │        终止           │
       └──────────────────────┘
```

**关键区别**：
- 就绪 → 运行：被调度器选中，获得 CPU
- 运行 → 就绪：时间片到了，被抢走 CPU
- 运行 → 阻塞：主动等 I/O（自己放弃 CPU）
- 阻塞 → 就绪：I/O 完成（不是直接回到运行态）

**面试话术**："进程三态模型是最基本的，六态模型加了挂起态。关键要理解：运行→就绪是被动的（时间片到），运行→阻塞是主动的（等 I/O）。阻塞态的进程不参与 CPU 竞争。"

### PCB（进程控制块）

操作系统用 PCB 来管理每个进程，它是一个数据结构，包含：
- **进程 ID（PID）**
- **进程状态**（就绪/运行/阻塞）
- **程序计数器 PC**：下一条要执行的指令地址
- **CPU 寄存器值**：切换时保存现场
- **内存管理信息**：页表地址
- **打开文件列表**：文件描述符表
- **父进程 PID**

```bash
# Linux 中查看进程信息
ps aux              # 查看所有进程
cat /proc/1234/status  # 查看 PID 1234 的详细状态
```

## 三、进程同步与互斥

### 临界区问题

多个线程同时访问共享资源会出问题：

```python
import threading

balance = 100
lock = threading.Lock()

# 有问题的版本
def withdraw_unprotected(amount):
    global balance
    if balance >= amount:
        # 这里如果有线程切换，另一个线程也可能通过 if 检查
        balance -= amount  # 两次减可能导致余额变负

# 正确的版本
def withdraw_safe(amount):
    global balance
    with lock:  # 获取锁，确保同一时刻只有一个线程进入
        if balance >= amount:
            balance -= amount
```

### 同步工具对比

| 机制 | 用途 | 特点 |
|------|------|------|
| 互斥锁（Mutex） | 保证同一时刻只有一个线程访问资源 | 最简单，二元状态 |
| 信号量（Semaphore） | 控制同时访问资源的线程数 | 计数器，可以允许多个 |
| 条件变量 | 线程间通知：某个条件满足了 | 配合锁使用，wait/notify |
| 读写锁 | 读可并发，写互斥 | 适合读多写少的场景 |

```python
import threading
import time

# 信号量示例：限制并发数
semaphore = threading.Semaphore(3)  # 最多允许3个线程同时

def task(n):
    with semaphore:
        print(f"任务 {n} 开始")
        time.sleep(1)
        print(f"任务 {n} 结束")

# 同时启动10个任务，但最多3个在运行
for i in range(10):
    threading.Thread(target=task, args=(i,)).start()
```

### 死锁（Deadlock）

**四个必要条件**（缺一个就不会死锁）：

| 条件 | 含义 | 打破方法 |
|------|------|----------|
| 互斥 | 资源不能共享 | 让资源可共享（不是所有资源能做到） |
| 持有并等待 | 拿着一个资源还在等另一个 | 一次性申请所有资源 |
| 不可剥夺 | 不能抢别人的资源 | 允许系统回收资源 |
| 环路等待 | A等B，B等C，C等A形成环 | 按固定顺序申请资源 |

```python
# 死锁演示
import threading
lock_a = threading.Lock()
lock_b = threading.Lock()

def thread1():
    with lock_a:
        time.sleep(0.1)  # 模拟处理
        with lock_b:      # 等 lock_b，但 thread2 拿着
            print("Thread 1 完成")

def thread2():
    with lock_b:
        time.sleep(0.1)
        with lock_a:      # 等 lock_a，但 thread1 拿着
            print("Thread 2 完成")

# Thread1: 拿A等B → Thread2: 拿B等A → 死锁！
# 解决方法：统一加锁顺序
# 两个线程都先拿 lock_a 再拿 lock_b 就不会死锁
```

**面试话术**："死锁需要四个条件同时满足：互斥、持有并等待、不可剥夺、环路等待。银行家算法通过检查资源分配后系统是否仍处于安全状态来避免死锁。工程上最实用的方法是按固定顺序加锁。"

### 经典同步问题

**生产者-消费者**：

```python
import threading
import queue
import time

# Python 的 queue.Queue 是线程安全的，内置了同步
q = queue.Queue(maxsize=10)

def producer():
    for i in range(20):
        q.put(i)
        print(f"生产: {i}")
        time.sleep(0.1)

def consumer():
    while True:
        item = q.get()
        print(f"消费: {item}")
        q.task_done()
        time.sleep(0.3)

threading.Thread(target=producer).start()
threading.Thread(target=consumer, daemon=True).start()
time.sleep(5)
```

## 四、内存管理

### 虚拟内存

**核心思想**：让每个进程以为自己独占一整块连续内存，实际上物理内存可能被切成碎块分配给多个进程。

```
程序A的虚拟地址空间            物理内存
┌──────────────┐            ┌─────────┐
│   代码段      │──┐         │  空      │
├──────────────┤  │    ┌───→├─────────┤
│   数据段      │  │    │    │ B的数据   │
├──────────────┤  │    │    ├─────────┤
│   堆          │──┤    │    │ A的代码   │
│              │  │    │    ├─────────┤
│     ↓        │  │    │    │ A的数据   │
│              │  │    │    ├─────────┤
│     ↑        │  │    │    │ B的代码   │
│              │  │    │    ├─────────┤
│   栈          │  │    │    │  A的堆    │
└──────────────┘  │    │    ├─────────┤
程序B的虚拟地址空间 │    │    │  B的堆    │
┌──────────────┐  │    │    ├─────────┤
│   代码段      │──┘    │    │  空      │
├──────────────┤       │    ├─────────┤
│   数据段      │───────┘    │  A的栈    │
├──────────────┤            ├─────────┤
│   ...        │            │  B的栈    │
└──────────────┘            └─────────┘
```

**虚拟地址 → 物理地址的转换**：MMU（内存管理单元）硬件完成，依赖操作系统维护的页表。

### 分页（Paging）

将物理内存和虚拟内存都切成固定大小的块——**页**（Page，通常 4KB）。

```
虚拟地址结构（32位，4KB页）：
┌──────────────┬──────────────┐
│   页号 (20位)  │ 页内偏移 (12位) │    2^12 = 4096 = 4KB
└──────────────┴──────────────┘

转换过程：
页号 → 页表查找 → 物理页框号 → 物理地址 = 页框号 × 4KB + 偏移
```

```bash
# Linux 查看页大小
getconf PAGE_SIZE  # 通常输出 4096
```

### 页面置换算法

当物理内存满了，需要换出一些页到磁盘（swap）。换哪个？

| 算法 | 策略 | 特点 |
|------|------|------|
| OPT（最优） | 换掉最晚被用到的页 | 理论最优，但无法实现（不知道未来） |
| FIFO | 换掉最早进来的页 | 简单但可能换掉常用页（Belady异常） |
| LRU（最近最久未使用） | 换掉最长时间没被访问的页 | 效果好，接近OPT，但实现开销大 |
| Clock（时钟算法） | 近似LRU，用访问位转圈 | Linux 实际使用，性价比高 |

**Belady 异常**：增加物理页框数，FIFO 算法的缺页次数反而增加——这不合直觉但确实会发生。LRU 不会出现 Belady 异常。

**面试话术**："LRU 是面试最常考的页面置换算法。核心思想是局部性原理——最近用过的页很可能马上再用。实现可以用链表+哈希表 O(1) 访问和移动，经典题是 LeetCode 146 LRU 缓存。"

### 分段（Segmentation）

分段是按**逻辑意义**切分：代码段、数据段、堆、栈各是一个段。每个段大小不同。

| 维度 | 分页 | 分段 |
|------|------|------|
| 划分依据 | 物理上，固定大小 | 逻辑上，按程序结构 |
| 大小 | 固定（4KB） | 不固定 |
| 程序员可见 | 不可见（透明） | 可见（代码段、数据段） |
| 碎片问题 | 内部碎片 | 外部碎片 |
| 共享/保护 | 按页（粒度粗） | 按逻辑单元（自然） |

现代操作系统通常把分页和分段结合——**段页式**。

## 五、文件系统

### 文件系统的基本概念

文件系统解决的根本问题：**如何把文件存到磁盘上，并快速找到它**。

关键概念：
- **inode（索引节点）**：存储文件的元数据——大小、权限、所有者、时间戳、数据块位置。但不存文件名。
- **目录**：本质上是一个特殊文件，内容是"文件名 → inode 号"的映射表。
- **数据块**：文件的实际内容存在多个数据块中。

```
inode 结构（简化）：
┌─────────────────────┐
│ 文件类型/权限        │
│ 硬链接数             │
│ 所有者 UID / GID     │
│ 文件大小             │
│ 时间戳（访问/修改/变更）│
│ 数据块指针[12个直接]   │──→ 数据块
│ 间接指针             │──→ 间接块 → 数据块
│ 双重间接指针          │──→ 二级间接块 → ...
│ 三重间接指针          │──→ 三级间接块 → ...
└─────────────────────┘
```

### 常见文件系统对比

| 文件系统 | 特点 | 典型场景 |
|----------|------|----------|
| ext4 | Linux 主流，支持日志，稳定 | Linux 桌面/服务器 |
| XFS | 高性能，支持大文件 | 数据库服务器 |
| NTFS | Windows 原生，支持权限、加密 | Windows 系统盘 |
| FAT32/exFAT | 兼容性最好，但不支持权限 | U盘、移动硬盘 |
| APFS | Apple 原生，优化 SSD | macOS |

### 硬链接 vs 软链接

| | 硬链接 | 软链接（符号链接） |
|------|--------|-------------------|
| 本质 | 同一个 inode 的两个名字 | 记录目标路径的"快捷方式" |
| 跨文件系统 | 不可以（共享 inode） | 可以 |
| 删原文件 | 不影响（只要至少一个硬链接在） | 软链接变死链 |
| 创建命令 | `ln file hardlink` | `ln -s file softlink` |

```bash
touch original.txt
ln original.txt hardlink.txt      # 硬链接：同一个 inode
ln -s original.txt softlink.txt   # 软链接：指向路径

ls -li   # -i 显示 inode 号，硬链接和原文件 inode 号相同
rm original.txt
# hardlink.txt 仍然能读（实际数据还在）
# softlink.txt 变成死链（指向的路径不存在了）
```

## 六、CPU 调度算法

### 常见调度算法

| 算法 | 策略 | 优点 | 缺点 |
|------|------|------|------|
| FCFS（先来先服务） | 按到达顺序 | 简单公平 | 长任务堵死短任务（护航效应） |
| SJF（最短作业优先） | 先完成最短的任务 | 平均等待时间最小 | 需要预知任务时长，长任务可能饿死 |
| RR（时间片轮转） | 轮流每人跑一小段 | 公平，响应快 | 时间片太小导致切换开销大 |
| 优先级调度 | 按优先级排序 | 重要任务优先 | 低优先级可能饿死 |
| 多级反馈队列 | 结合上述所有 | 实际 OS 使用，兼顾一切 | 参数调优复杂 |

**多级反馈队列的工作原理**：
1. 新进程进入最高优先级队列（时间片小）
2. 用完时间片还不结束 → 降到下一级队列（时间片翻倍）
3. 始终优先运行高优先级队列
4. 同一个队列内用 RR

这样**短作业在高优先级队列很快就完成，长作业逐渐降级，CPU 密集和 I/O 密集都能兼顾**。

### 调度性能指标

| 指标 | 含义 | 关心这个指标的场景 |
|------|------|-------------------|
| CPU 利用率 | CPU 忙着的时间占比 | 服务器 |
| 吞吐量 | 单位时间完成的任务数 | 批处理 |
| 周转时间 | 从提交到完成的时间 | 批处理 |
| 等待时间 | 在就绪队列中等待的时间 | 用户体验 |
| 响应时间 | 从提交到第一次响应的时间 | 交互式系统 |

**面试话术**："没有哪个调度算法是万能的。交互式系统追求响应快（用 RR 或优先级），批处理系统追求吞吐量（用 SJF）。现代 Linux 用的是 CFS（完全公平调度器），给每个进程按权重分配 CPU 时间比例。"

## 七、I/O 管理

### I/O 控制方式

| 方式 | 工作机制 | 特点 |
|------|----------|------|
| 程序直接控制 | CPU 不断轮询检查设备状态 | CPU 全程占用，浪费 |
| 中断驱动 | I/O 完成时设备发中断通知 CPU | CPU 不用等，效率提升 |
| DMA（直接存储器访问） | 专门硬件控制器，绕过 CPU 直接传输 | CPU 只要开头结尾参与，效率最高 |

### 磁盘调度算法

| 算法 | 策略 | 特点 |
|------|------|------|
| FCFS | 按请求顺序 | 公平但效率低 |
| SSTF（最短寻道时间优先） | 先处理离当前磁头最近的请求 | 效率高，但远处请求可能饿死 |
| SCAN（电梯算法） | 磁头来回扫，沿途处理 | 公平且高效，实际常用 |
| C-SCAN | 单向扫到底，快速归位再重新扫 | 更均匀的响应时间 |

## 八、面试高频考点速查

| 考点 | 出题概率 | 必答要点 |
|------|----------|----------|
| 进程 vs 线程 | 极高 | 资源基本单位 vs 调度基本单位、共享 vs 隔离、开销对比 |
| 进程状态转换 | 高 | 三态/六态图、就绪→运行（被调度）、运行→阻塞（主动等I/O） |
| 死锁 | 极高 | 四个必要条件 + 各条件的打破方法 |
| 虚拟内存 | 较高 | 逻辑地址→物理地址、页表、MMU、缺页中断 |
| 分页 vs 分段 | 中 | 固定大小 vs 逻辑划分、内碎片 vs 外碎片 |
| LRU 算法 | 高 | 最近最久未使用、局部性原理、实现方式 |
| 硬链接 vs 软链接 | 中 | 同inode共享 vs 路径快捷方式、跨文件系统 |
| 时间片轮转 | 中 | 公平、响应快、时间片大小权衡 |
