# 软件工程

> 参考教材：《软件工程案例教程（第4版）》。软件工程不是教你写代码，而是教你**怎么有组织地把代码写对、写好、写到位**。校招面试考得不多，但社招和系统设计会大量涉及。

## 一、软件生命周期

软件生命周期描述软件从"想法"到"退役"的完整过程：

```
┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
│ 需求  │→│ 设计  │→│ 编码  │→│ 测试  │→│ 部署  │→│ 维护  │
│分析  │  │      │  │      │  │      │  │      │  │      │
└──────┘  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘
```

### 三大开发模型

| 模型 | 流程 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| 瀑布 | 需求→设计→编码→测试→运维，严格串行 | 清晰、文档全 | 不能回头、需求变一点就崩 | 需求极其稳定（如军工、航天） |
| 敏捷（Scrum） | 2-4周迭代，每个迭代都交付可用软件 | 灵活、反馈快 | 文档弱、大型团队难协调 | 互联网产品、需求频繁变化 |
| DevOps | 开发+运维一体化，持续集成部署 | 交付极快、质量反馈闭环 | 对自动化要求高、文化变革难 | SaaS 产品、微服务架构 |

### 瀑布 vs 敏捷（面试最爱考）

| 维度 | 瀑布 | 敏捷（Scrum） |
|------|------|--------------|
| 需求 | 一开始就锁死 | 每个迭代都可以变 |
| 交付 | 项目结束时一次性交付 | 每 2-4 周交付可用增量 |
| 文档 | 重视，每个阶段产出文档 | 够用就行，可用的软件 > 文档 |
| 变更 | 越晚越贵，尽量避免 | 拥抱变化 |
| 客户参与 | 开始和结束时参与 | 全程参与，每个 Sprint Review |
| 团队 | 按角色分工 | 自组织跨职能小团队 |
| 度量 | 按计划完成度 | 按实际交付价值 |

**Scrum 核心角色和仪式**：

| 角色 | 职责 |
|------|------|
| Product Owner（产品负责人） | 定义需求优先级，维护 Product Backlog |
| Scrum Master | 保障 Scrum 流程，消除障碍（不是项目经理） |
| Development Team | 自组织完成开发（5-9人） |

| 仪式 | 做什么 | 时长 |
|------|--------|------|
| Sprint Planning | 确定本次 Sprint 要做的任务 | 每 Sprint 开始，2-4小时 |
| Daily Standup | 每人回答：昨天做了啥、今天做啥、有什么障碍 | 每天 15 分钟 |
| Sprint Review | 给 Stakeholder 演示交付成果 | 每 Sprint 结束，1-2 小时 |
| Sprint Retrospective | 团队内部复盘：什么做得好、什么要改进 | 每 Sprint 结束，1-1.5 小时 |

**面试话术**："敏捷不是更快地写代码，而是更快地获得反馈。Scrum 的 2-4 周 Sprint 确保每两周就能验证方向对不对，而不是写完半年才发现做偏了。现在大部分互联网公司是 Scrum 框架 + DevOps 工具链的混合模式。"

### DevOps 核心理念

```
传统模式：Dev 写完 → 扔给 Ops → Ops 部署 → 出问题 Dev 说"我代码没问题"...
DevOps：  Dev/Ops 同一个团队 → CI/CD 自动测试部署 → 共享责任
```

关键实践：
- **CI（持续集成）**：代码提交后自动构建 + 跑测试
- **CD（持续交付/部署）**：通过测试后自动部署到生产环境
- **IaC（基础设施即代码）**：用代码管理服务器配置（Terraform, Ansible）
- **监控和日志**：全链路追踪、告警

## 二、需求分析

### 用户故事（User Story）

格式：**作为 \<角色\>，我想要 \<功能\>，以便 \<价值\>**。

```
好的用户故事：
"作为普通用户，我想要按日期筛选订单，以便快速找到上个月的消费记录。"

不好的用户故事：
"系统需要支持订单筛选功能。"  ← 没有说谁用、为什么用
```

**INVEST 原则**（好用户故事的六个标准）：

| 字母 | 含义 | 检查 |
|------|------|------|
| I | Independent（独立） | 能独立开发，不依赖其他故事 |
| N | Negotiable（可协商） | 细节可以讨论，不是死需求 |
| V | Valuable（有价值） | 对用户/业务有明确价值 |
| E | Estimable（可估算） | 团队能估算工作量 |
| S | Small（小） | 一个 Sprint 内能完成 |
| T | Testable（可测试） | 有明确的完成标准 |

### 用例图（Use Case Diagram）

用例图描述"谁（Actor）能用系统做什么（Use Case）"：

```
                    ┌─────────────┐
                    │  在线商城系统  │
                    │             │
    ┌───────┐       │  ┌─────────┐│
    │ 普通用户├────────→│ 浏览商品  ││
    └───┬───┘       │  └────┬────┘│
        │           │       │     │
        │ 继承       │  ┌────▼────┐│
    ┌───▼───┐       │  │ 下单购买  ││←───┐
    │  VIP用户│      │  └────┬────┘│    │
    └───────┘       │       │     │    扩展
                    │  ┌────▼────┐│    │
    ┌───────┐       │  │ 积分抵扣  ││────┘
    │ 管理员  ├──────→│  └─────────┘│
    └───────┘       │  ┌─────────┐│
                    │  │ 管理商品  ││
                    │  └─────────┘│
                    └─────────────┘
```

基本关系：
- **关联**：Actor 和 Use Case 之间的连线，表示"执行"
- **包含（include）**：虚线箭头 + `<<include>>`，A 必然调用 B（下单 include 支付）
- **扩展（extend）**：虚线箭头 + `<<extend>>`，B 可选地扩展 A（积分抵扣 extend 下单）
- **继承**：Actor 之间的三角箭头，VIP 用户继承普通用户

## 三、系统设计

### 常用架构模式

| 模式 | 核心思想 | 优点 | 缺点 | 场景 |
|------|----------|------|------|------|
| 分层架构 | Controller→Service→DAO 三层分离 | 简单、理解成本低 | 跨层调用变多时耦合重 | 中小型 Web 应用 |
| MVC | Model（数据）+ View（界面）+ Controller（逻辑） | 关注点分离 | Controller 容易臃肿 | Web 应用、移动端 |
| 微服务 | 拆成独立小服务，独立部署 | 独立扩展、故障隔离 | 分布式复杂度（网络、数据一致性） | 大型系统 |
| 事件驱动 | 通过事件/消息通信，解耦生产者和消费者 | 松耦合、易扩展 | 调试困难、最终一致性 | 异步处理、消息队列 |

### MVC 详解

```
用户请求
    │
    ▼
┌──────────────┐     ┌──────────────┐
│   Controller  │────→│    Model     │
│   (控制器)     │     │   (数据层)    │
│               │◄────│              │
│ Spring @Cont- │     │ 业务逻辑      │
│ roller /      │     │ 数据库操作     │
│ Django Views  │     │              │
└──────┬───────┘     └──────────────┘
       │
       │ 传递数据
       ▼
┌──────────────┐
│    View       │
│   (视图层)     │
│               │
│ HTML/模板/    │
│ React 组件    │
└──────────────┘
```

```python
# Django 风格的 MVC（Django 叫 MVT：Model-View-Template）
# Model
class Order(models.Model):
    user_id = models.IntegerField()
    amount = models.DecimalField(max_digits=10, decimal_places=2)
    created_at = models.DateTimeField(auto_now_add=True)

# View (Controller)
def order_list(request):
    orders = Order.objects.filter(user_id=request.user.id)
    return render(request, 'order_list.html', {'orders': orders})

# Template (View)
# <ul>
# {% for order in orders %}
#   <li>{{ order.amount }} - {{ order.created_at }}</li>
# {% endfor %}
# </ul>
```

### 微服务核心概念

```
单体架构                    微服务架构
┌──────────────────┐      ┌─────┐ ┌─────┐ ┌─────┐
│  用户模块          │      │用户  │ │订单  │ │商品  │
│  订单模块          │  →   │服务  │ │服务  │ │服务  │
│  商品模块          │      └──┬──┘ └──┬──┘ └──┬──┘
│  支付模块          │        │       │       │
└──────────────────┘      ┌──┴───────┴───────┴──┐
  一个进程、一个数据库       │  各自独立数据库/进程   │
                          └────────────────────┘
```

微服务的关键挑战：
- **服务发现**：服务 A 怎么知道服务 B 的地址？（Consul, Eureka, K8s Service）
- **配置管理**：几十个服务的配置怎么统一管理？（配置中心）
- **服务间通信**：同步（REST/gRPC）还是异步（消息队列）？
- **分布式事务**：跨多个服务的数据一致性？（Saga 模式、最终一致）
- **可观测性**：请求跨了 3 个服务，怎么追踪？（分布式链路追踪）

**面试话术**："微服务不是银弹。团队小于 10 人、系统不复杂的时候，单体架构可能更合适。微服务的核心代价是分布式系统的复杂度——网络不可靠、数据一致性难保证、调试排障成本高。"

## 四、UML 基础

UML（统一建模语言）是用图来表达软件设计。面试最多问到类图和时序图。

### 类图（Class Diagram）

```
┌──────────────────────────┐
│        <<abstract>>       │
│          Animal           │  类名（抽象类用斜体或 <<abstract>>）
├──────────────────────────┤
│ - name: String           │  属性（- 私有, + 公有, # 保护）
│ - age: int               │
├──────────────────────────┤
│ + eat(): void            │  方法
│ + sleep(): void          │
│ + makeSound(): void      │  抽象方法用斜体
└──────────┬───────────────┘
           │ 继承（空心三角箭头）
    ┌──────┴──────┐
    │              │
┌───┴──────┐  ┌───┴──────┐
│   Dog    │  │   Cat    │
├──────────┤  ├──────────┤
│ + bark() │  │ + meow() │
└──────────┘  └─────┬────┘
                     │
                     │ 关联（实线箭头）
                     │
              ┌──────┴──────┐
              │   Owner     │
              ├─────────────┤
              │ + name      │
              │ + feed()    │
              └─────────────┘
```

**常用关系**：

| 关系 | 符号 | 含义 | 代码体现 |
|------|------|------|----------|
| 继承（泛化） | 空心三角实线 | "is-a" | `class Dog extends Animal` |
| 实现 | 空心三角虚线 | "implements" | `class ServiceImpl implements IService` |
| 关联 | 实线箭头 | "uses/has" | 一个类持有另一个类的引用 |
| 聚合 | 空心菱形实线 | 整体-部分，部分可独立存在 | `List<Student>` 在 School 中 |
| 组合 | 实心菱形实线 | 整体-部分，同生共死 | `Engine` 是 `Car` 的组成部分 |
| 依赖 | 虚线箭头 | 临时使用 | 方法的参数类型 |

**面试速记**：记住两个最常考的——**继承是三角箭头**，**组合比聚合更强**（组合同生共死，聚合各过各的）。

## 五、设计原则

### SOLID 五原则

| 原则 | 全称 | 一句话解释 |
|------|------|-----------|
| S | 单一职责（Single Responsibility） | 一个类只干一件事 |
| O | 开闭原则（Open/Closed） | 对扩展开放，对修改关闭 |
| L | 里氏替换（Liskov Substitution） | 子类能替换父类，不出问题 |
| I | 接口隔离（Interface Segregation） | 别强迫客户依赖它不用的接口 |
| D | 依赖倒置（Dependency Inversion） | 依赖抽象，不依赖具体实现 |

```python
# SOLID 示例

# ❌ 违反单一职责：一个类既处理业务又处理数据库
class ReportGenerator:
    def generate(self, data):
        # 格式化成 PDF
        pass
    def save_to_db(self, report):
        # 存数据库
        pass

# ✅ 拆分职责
class ReportGenerator:
    def generate(self, data):
        pass

class ReportRepository:
    def save(self, report):
        pass

# ❌ 违反开闭：加新图表类型要改 if-else
def draw_chart(chart_type, data):
    if chart_type == "bar":
        return draw_bar(data)
    elif chart_type == "pie":
        return draw_pie(data)
    # 加新类型要改这里

# ✅ 对扩展开放：新增图表类型只需加新类
class Chart:
    def draw(self, data):
        raise NotImplementedError

class BarChart(Chart):
    def draw(self, data):
        pass

class PieChart(Chart):
    def draw(self, data):
        pass

# ❌ 违反依赖倒置：高层依赖低层具体实现
class OrderService:
    def __init__(self):
        self.db = MySQLDatabase()  # 直接依赖具体类

# ✅ 依赖接口：可以随时换数据库
class OrderService:
    def __init__(self, db: Database):  # 依赖抽象
        self.db = db
```

### DRY vs KISS vs YAGNI

| 原则 | 含义 | 反例 |
|------|------|------|
| DRY（Don't Repeat Yourself） | 不要重复 | 三处复制粘贴同一段逻辑 |
| KISS（Keep It Simple, Stupid） | 保持简单 | 为了"将来可能需要"写一堆抽象 |
| YAGNI（You Aren't Gonna Need It） | 你不需要它 | 老板没要求的功能不要加 |

```python
# DRY 错误示范
def validate_email(email):
    if "@" not in email:
        raise ValueError("邮箱格式错误")
    return email

def validate_admin_email(email):
    if "@" not in email:          # 重复逻辑
        raise ValueError("邮箱格式错误")
    if not email.endswith("@company.com"):
        raise ValueError("需要公司邮箱")
    return email

# DRY 正确做法
def validate_email(email):
    if "@" not in email:
        raise ValueError("邮箱格式错误")
    return email

def validate_admin_email(email):
    validate_email(email)  # 复用
    if not email.endswith("@company.com"):
        raise ValueError("需要公司邮箱")
    return email
```

**面试话术**："SOLID 不是教条，是经验法则。实际写代码时，单一职责和开闭原则最实用。DRY 是最高频的错误——看到复制粘贴就警惕。但也要注意 KISS——不要为了 DRY 把两个只是碰巧相似的逻辑强行统一。"

## 六、测试

### 测试金字塔

```
        ╱  E2E 测试 ╲        少量，慢，覆盖核心流程
      ╱───────────────╲
     ╱   集成测试       ╲      中量，中速，验证模块间交互
    ╱───────────────────╲
   ╱    单元测试          ╲    大量，快速，验证单个函数/方法
  ╱─────────────────────────╲
```

### 单元测试

```python
# 被测试的代码
def calculate_discount(amount: float, is_vip: bool) -> float:
    """计算折扣后的价格"""
    if amount <= 0:
        raise ValueError("金额必须大于0")

    if is_vip:
        if amount >= 1000:
            return amount * 0.7   # VIP + 大额 = 7折
        return amount * 0.85       # VIP = 85折
    else:
        if amount >= 1000:
            return amount * 0.9    # 大额 = 9折
        return amount               # 原价

# 单元测试（pytest）
import pytest

class TestCalculateDiscount:
    def test_vip_large_amount(self):
        assert calculate_discount(1000, True) == 700

    def test_vip_small_amount(self):
        assert calculate_discount(100, True) == 85

    def test_regular_large_amount(self):
        assert calculate_discount(1000, False) == 900

    def test_regular_small_amount(self):
        assert calculate_discount(100, False) == 100

    def test_boundary_vip_999(self):
        """边界测试：刚好不到 1000"""
        assert calculate_discount(999, True) == 849.15

    def test_invalid_amount_raises(self):
        with pytest.raises(ValueError):
            calculate_discount(0, False)

    def test_negative_amount_raises(self):
        with pytest.raises(ValueError):
            calculate_discount(-100, True)

# 运行: pytest test_discount.py -v
```

**边界条件要测什么**：0、负数、恰好等于阈值（999 vs 1000）、极大值。

### TDD（测试驱动开发）

三步循环：**Red → Green → Refactor**

1. **Red**：先写一个失败的测试
2. **Green**：写最少的代码让测试通过
3. **Refactor**：重构代码，保持测试通过

```python
# TDD 示例——开发一个"判断是否为回文"的函数

# Step 1: Red — 先写测试（代码还不存在）
def test_is_palindrome():
    assert is_palindrome("racecar") == True
    assert is_palindrome("hello") == False
    assert is_palindrome("") == True  # 空字符串算回文
    assert is_palindrome("A man a plan a canal Panama") == True

# Step 2: Green — 写最简代码让测试通过
def is_palindrome(s: str) -> bool:
    cleaned = ''.join(c.lower() for c in s if c.isalnum())
    return cleaned == cleaned[::-1]

# Step 3: Refactor — 可能改成双指针提高空间效率
def is_palindrome(s: str) -> bool:
    left, right = 0, len(s) - 1
    while left < right:
        while left < right and not s[left].isalnum():
            left += 1
        while left < right and not s[right].isalnum():
            right -= 1
        if s[left].lower() != s[right].lower():
            return False
        left += 1
        right -= 1
    return True
```

**面试话术**："TDD 的价值不是测试本身，而是强迫你从使用者的角度思考接口设计。先写测试会让你自然地问'调用方会怎么用这个函数'，从而设计出更好的 API。实际工作中不一定严格 TDD，但关键逻辑一定有测试。"

### 测试类型对比

| 测试类型 | 范围 | 速度 | 维护成本 | 执行频率 |
|----------|------|------|----------|----------|
| 单元测试 | 单个函数/方法 | 毫秒级 | 低 | 每次提交 |
| 集成测试 | 多个模块交互 | 秒级 | 中 | 每次提交 |
| E2E 测试 | 完整用户流程 | 分钟级 | 高 | 每日/每次发布 |
| 性能测试 | 系统负载 | 分钟-小时 | 高 | 重大变更前 |

## 七、版本控制与 CI/CD

> 详见 [[工作学习/工程基础/Git基础与协作|Git 基础与协作]]

### CI/CD 流水线

```
开发者 push 代码
       │
       ▼
  ┌─────────────────────┐
  │   CI (持续集成)       │
  │  1. 拉取代码          │
  │  2. 安装依赖          │
  │  3. 运行 lint         │
  │  4. 运行单元测试       │
  │  5. 构建镜像          │
  └─────────┬───────────┘
            │ 全部通过
            ▼
  ┌─────────────────────┐
  │   CD (持续部署)       │
  │  1. 部署到测试环境     │
  │  2. 运行集成/E2E测试   │
  │  3. 灰度发布          │
  │  4. 全量发布          │
  └─────────────────────┘
```

**常用工具链**：
- 代码仓库：GitHub / GitLab
- CI/CD：GitHub Actions / GitLab CI / Jenkins
- 容器化：Docker
- 编排：Kubernetes / Docker Compose
- 监控：Prometheus + Grafana / ELK

### Git 分支策略

```
main  ───●──────────────────●──────●── 生产代码
          \                /      /
release    ●──────●──────●       /
                  \            /
develop            ●──●──●──●──  开发主线
                        \    /
feature                  ●──●    功能分支
```

- **main**：生产环境代码，只从 release 合并
- **develop**：开发主线
- **feature/xxx**：新功能分支，从 develop 拉，合并回 develop
- **release/x.x.x**：发布准备分支，修 bug，合并到 main 和 develop
- **hotfix/xxx**：紧急修复，从 main 拉，合并回 main 和 develop

## 八、面试高频考点

| 考点 | 出题概率 | 必答要点 |
|------|----------|----------|
| 敏捷 vs 瀑布 | 高 | 迭代交付 vs 一次性交付、拥抱变化 vs 锁定需求 |
| SOLID 原则 | 高 | 至少说清 SRP 和 OCP |
| 设计模式（单例、工厂） | 中 | 能写代码、知道适用场景 |
| MVC 架构 | 中 | 三层职责、数据流向 |
| 单元测试 vs 集成测试 | 中 | 测试金字塔、各自范围 |
| CI/CD | 中 | 持续集成/持续部署的概念和价值 |
| 用户故事 | 低 | As a... I want... So that... 格式 |
| 微服务 | 中 | 优缺点、何时不该用 |

### 常见设计模式速记

| 模式 | 解决的问题 | 一句话 |
|------|-----------|--------|
| 单例（Singleton） | 全局只需要一个实例 | 配置管理、数据库连接池 |
| 工厂方法 | 隐藏对象创建的细节 | 根据类型参数创建不同对象 |
| 观察者 | 一对多的通知 | 事件监听、发布订阅 |
| 策略 | 算法可互换 | 不同的支付方式、排序策略 |
| 装饰器 | 不修改原类加功能 | Python 的 @staticmethod、日志包装 |
| 适配器 | 接口不兼容的类协作 | 新接口包装老代码 |
