## 一、计算机发展简史

### 四代计算机

| 代 | 年代 | 核心元件 | 特点 |
|----|------|----------|------|
| 第一代 | 1946-1958 | 电子管 | 笨重、耗电、速度慢（ENIAC 占满一间房） |
| 第二代 | 1958-1964 | 晶体管 | 体积缩小百倍，可靠性大幅提升 |
| 第三代 | 1964-1971 | 集成电路（IC） | 一块芯片集成多个晶体管，小型机出现 |
| 第四代 | 1971至今 | 超大规模集成电路（VLSI） | 单芯片集成百万/亿级晶体管，PC 普及 |

**面试话术**："计算机发展遵循摩尔定律——集成电路上的晶体管数量每18-24个月翻一番。但这个定律在物理极限下正在放缓。"

### 冯·诺依曼架构

现代计算机几乎都是冯·诺依曼架构，核心思想是**存储程序**：

```
┌──────────────────────────────────────┐
│              计算机系统               │
│  ┌────────┐  ┌────────┐  ┌────────┐ │
│  │ 运算器  │  │ 控制器  │  │ 存储器  │ │
│  │ (ALU)  │  │ (CU)   │  │(Memory)│ │
│  └───┬────┘  └───┬────┘  └───┬────┘ │
│      │           │            │      │
│      └───────────┼────────────┘      │
│                  │                   │
│            ┌─────┴─────┐             │
│            │  输入/输出  │             │
│            └───────────┘             │
└──────────────────────────────────────┘
```

五大部件：
1. **运算器（ALU）**：做算术和逻辑运算
2. **控制器（CU）**：指挥各部件协调工作
3. **存储器**：存放程序和数据
4. **输入设备**：键盘、鼠标、网卡
5. **输出设备**：显示器、打印机、网卡

**核心瓶颈**：CPU 和内存之间的速度差——CPU 很快（纳秒级），内存较慢（几十纳秒）。这就是为什么需要缓存（Cache）。

**面试话术**："冯·诺依曼架构的本质是'程序和数据都存在同一存储器中，按地址访问依次执行'。这个设计的瓶颈是CPU和内存之间的带宽墙，现代计算机用多级缓存和流水线来缓解。"

## 二、数据在计算机中的表示

### 进制转换

计算机只认识0和1。所有数据——数字、文字、图片、视频——最终都是二进制。

```
十进制  →  二进制  →  八进制  →  十六进制
  10        1010       12         A
  15        1111       17         F
  255     11111111      377       FF
```

**手算技巧**：
- 十进制转二进制：不断除以2取余数，倒着写
- 二进制转十六进制：每4位二进制 = 1位十六进制（如 `1111 1010` = `FA`）
- 十六进制转二进制：每位十六进制展开为4位二进制

```python
# Python 中的进制转换
n = 42
print(bin(n))   # '0b101010' — 二进制
print(oct(n))   # '0o52'     — 八进制
print(hex(n))   # '0x2a'     — 十六进制

# 字符串转整数
print(int('101010', 2))   # 42（二进制字符串转十进制）
print(int('FF', 16))      # 255（十六进制字符串转十进制）
```

### 补码（Two's Complement）

**为什么不直接用二进制表示负数？** 因为直接用最高位做符号位，`+0` 和 `-0` 会重复，而且加减法电路不一样。

补码的巧妙之处：**用加法电路实现减法**。

```
以8位为例，-1 的表示：
 1 的二进制：  00000001
 按位取反：    11111110
 加 1：      11111111  ← -1 的补码表示

验证 -1 + 1 = 0：
  11111111  (-1)
+ 00000001  (1)
────────────
 100000000  (进位溢出丢弃) → 00000000 ✓
```

| 补码（8位） | 十进制值 |
|-------------|----------|
| 01111111 | +127（最大正数） |
| 00000001 | +1 |
| 00000000 | 0 |
| 11111111 | -1 |
| 10000000 | -128（最小负数） |

**面试话术**："补码让减法变成加负数，而加负数可以用加法电路实现。硬件只需要加法器就够了，简化了CPU设计。补码的另一个好处是0只有一种表示。"

### 浮点数（IEEE 754）

浮点数的"浮"是指小数点的位置是浮动的。单精度32位结构：

```
┌──────┬──────────┬─────────────────────┐
│ 符号 │   指数    │       尾数           │
│ 1位  │   8位    │       23位           │
└──────┴──────────┴─────────────────────┘

值 = (-1)^符号 × 1.尾数 × 2^(指数-127)
```

**经典坑：0.1 + 0.2 ≠ 0.3**

```python
print(0.1 + 0.2)           # 0.30000000000000004
print(0.1 + 0.2 == 0.3)   # False
```

原因：0.1 和 0.2 在二进制中是无限循环小数，存储时被截断，加在一起就产生了微小误差。

**面试话术**："浮点数不是精确的。金融计算不能用 float，要用 decimal 类型或整数表示分。比较浮点数时用 `abs(a-b) < epsilon` 而不是 `==`。"

### 字符编码

| 编码 | 字节数 | 覆盖范围 |
|------|--------|----------|
| ASCII | 1字节 | 英文字母、数字、符号（128个） |
| GB2312/GBK | 2字节 | 中文字符 |
| UTF-8 | 1-4字节变长 | 全球所有文字，兼容 ASCII |
| UTF-16 | 2或4字节 | 全球文字 |

```python
# 编码/解码
s = "计算机"
b = s.encode('utf-8')     # b'\xe8\xae\xa1\xe7\xae\x97\xe6\x9c\xba'
print(b)
print(b.decode('utf-8'))   # '计算机'

# Unicode 码点
print(ord('计'))           # 35745 十进制
print(hex(ord('计')))      # 0x8ba1 十六进制
print(chr(0x8ba1))         # '计'
```

## 三、计算机硬件组成

### CPU（中央处理器）

CPU 是计算机的大脑，内部结构：

```
┌──────────────────────────────────┐
│              CPU                  │
│  ┌────────────────────────────┐  │
│  │        控制单元 (CU)        │  │
│  │  ┌──────────────────────┐  │  │
│  │  │     指令寄存器 IR     │  │  │
│  │  │     程序计数器 PC     │  │  │
│  │  └──────────────────────┘  │  │
│  └────────────────────────────┘  │
│  ┌────────────────────────────┐  │
│  │       运算单元 (ALU)       │  │
│  └────────────────────────────┘  │
│  ┌────────────────────────────┐  │
│  │        寄存器组             │  │
│  │   AX BX CX DX SP BP ...    │  │
│  └────────────────────────────┘  │
│  ┌──────────┐  ┌──────────┐     │
│  │  L1 缓存  │  │  L2 缓存  │     │
│  └──────────┘  └──────────┘     │
└──────────────────────────────────┘
```

**指令周期**（CPU 的工作节奏）：取指 → 译码 → 执行 → 写回，不停循环。

**关键指标**：
- **主频**：每秒多少时钟周期（GHz），代表"心跳"速度
- **核心数**：物理上有几个 CPU
- **线程数**：逻辑上能同时跑几个任务（超线程技术让一个核心模拟两个线程）

### 存储层次

越往上越快、越贵、越小：

```
速度 ▲        寄存器 (Register)      ← 纳秒级，CPU内部，几百字节
    │        L1/L2/L3 缓存 (Cache)   ← 几纳秒，MB级别
    │        主存 (RAM)              ← 几十纳秒，GB级别
    │        固态硬盘 (SSD)           ← 微秒级，几百GB
    │        机械硬盘 (HDD)           ← 毫秒级，TB级别
    └──────────────────────────────→ 容量
```

**为什么需要缓存？** CPU 太快，内存太慢。CPU 等内存一次的时间可以执行几十上百条指令。缓存在中间做缓冲，利用**局部性原理**（刚用过的数据很可能再用，附近的数据很可能马上用）。

**面试话术**："存储层次是计算机体系结构最核心的设计思想——用不同速度和容量的存储器组成金字塔，让程序看起来像是在用又大又快的存储器。"

## 四、操作系统基本概念

详见 [[工作学习/计算机基础/操作系统|操作系统]]，这里先讲最核心的概念。

操作系统的本质：**硬件资源和应用程序之间的中间层**。

四个核心功能：
1. **进程管理**：谁先跑、跑多久、怎么切换
2. **内存管理**：给谁分内存、怎么回收、防止打架
3. **文件管理**：数据怎么存到磁盘、怎么找到
4. **设备管理**：键盘、鼠标、打印机等 I/O 设备调度

**系统调用**：应用程序不能直接操作硬件，必须通过操作系统提供的"窗口"——系统调用来操作。比如 `open()` 读文件、`fork()` 创建进程。

```bash
# Linux 中查看系统调用
strace ls        # 追踪 ls 命令的所有系统调用
```

## 五、网络基础

### OSI 七层模型 vs TCP/IP 四层模型

| OSI 七层 | TCP/IP 四层 | 作用 | 典型协议/设备 |
|----------|-------------|------|--------------|
| 应用层 | 应用层 | 为用户提供服务 | HTTP, DNS, SMTP, FTP |
| 表示层 | 应用层 | 数据格式转换、加密 | TLS/SSL |
| 会话层 | 应用层 | 建立/管理/终止会话 | RPC |
| 传输层 | 传输层 | 端到端可靠传输 | TCP, UDP |
| 网络层 | 网络层 | 寻址和路由选择 | IP, 路由器 |
| 数据链路层 | 网络接口层 | 相邻节点间传输 | MAC, 交换机 |
| 物理层 | 网络接口层 | 比特流传输 | 网线、光纤、集线器 |

**面试话术**："OSI 是理论模型，实际用的是 TCP/IP。面试不会让你背七层，但要能说清楚：应用层管'做什么'（HTTP），传输层管'怎么传'（TCP/UDP），网络层管'往哪传'（IP地址路由）。"

### TCP vs UDP

| | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接（三次握手） | 无连接 |
| 可靠性 | 可靠，保证到达，丢包重传 | 不可靠，发了不管 |
| 顺序 | 保证顺序 | 不保证 |
| 速度 | 慢 | 快 |
| 场景 | HTTP、文件传输、邮件 | 视频直播、游戏、DNS |

**经典面试题："浏览器输入 URL 后发生了什么？"**

答：DNS 解析域名 → TCP 三次握手建立连接 → 发送 HTTP 请求 → 服务器处理返回响应 → 浏览器解析 HTML/CSS/JS → 渲染页面。涉及 DNS(UDP)、TCP、HTTP、HTML 解析等多个层。

## 六、算法与数据结构引论

> 详见 [[工作学习/数据结构与算法/数组与双指针|数组与双指针]]、[[工作学习/数据结构与算法/哈希表与查找|哈希表与查找]]、[[工作学习/数据结构与算法/栈与队列|栈与队列]]

### 什么是算法

算法就是解决问题的步骤集合。算法的五个特性：
- **有穷性**：有限步后结束
- **确定性**：每步明确无歧义
- **可行性**：每步可执行
- **输入**：零个或多个
- **输出**：至少一个

### 时间复杂度速览

| 复杂度 | 名称 | 典型算法 | 数据量100万时 |
|--------|------|----------|--------------|
| O(1) | 常数 | 哈希查找 | 1步 |
| O(log n) | 对数 | 二分查找 | 约20步 |
| O(n) | 线性 | 遍历数组 | 100万步 |
| O(n log n) | 线性对数 | 归并排序 | 约2000万步 |
| O(n²) | 平方 | 冒泡排序 | 1万亿步（跑不动） |
| O(2ⁿ) | 指数 | 暴力子集 | 宇宙灭亡也算不完 |

**面试话术**："分析复杂度时看最坏情况和数据规模。10²级别 O(n²) 可以接受，10⁶级别 O(n²) 就崩了。工程上 O(n log n) 和 O(n) 在实际数据量下差距不一定大，常数因子也很重要。"

### 基本数据结构

| 结构 | 特点 | 查找 | 插入 | 删除 |
|------|------|------|------|------|
| 数组 | 连续内存，支持随机访问 | O(1) 按索引 | O(n) | O(n) |
| 链表 | 不连续，通过指针连接 | O(n) | O(1) 头插 | O(1) |
| 栈 | 后进先出（LIFO） | - | O(1) push | O(1) pop |
| 队列 | 先进先出（FIFO） | - | O(1) 入队 | O(1) 出队 |
| 哈希表 | 键值对，O(1)查找 | O(1) 平均 | O(1) 平均 | O(1) 平均 |
| 树 | 分层结构 | O(log n) 平衡 | O(log n) 平衡 | O(log n) 平衡 |

## 七、编程语言分类

### 编译型 vs 解释型

| | 编译型 | 解释型 |
|------|--------|--------|
| 执行方式 | 源代码 → 编译器 → 机器码 → 执行 | 源代码 → 解释器逐行执行 |
| 速度 | 快（直接是机器码） | 慢（边翻译边执行） |
| 调试 | 需要先编译 | 改了就能跑 |
| 典型语言 | C, C++, Go, Rust | Python, JavaScript, Ruby |
| 移植性 | 不同平台需重新编译 | 有解释器就能跑 |

**Java 是个特例**：先编译成字节码（.class），再由 JVM 解释/即时编译（JIT）执行。这是"半编译半解释"。

### 语言范式

| 范式 | 核心思想 | 代表语言 |
|------|----------|----------|
| 过程式 | 按步骤执行，函数组织代码 | C |
| 面向对象 | 数据和操作封装成对象 | Java, C++, Python |
| 函数式 | 函数是一等公民，无副作用 | Haskell, Scala, JS(部分) |
| 声明式 | 描述"要什么"而非"怎么做" | SQL, HTML |

#### 通俗类比

| 范式 | 类比 |
|------|------|
| 过程式 | 菜谱：第一步洗菜、第二步切菜、第三步下锅 |
| OOP | 乐高：每个零件有自己的形状和功能，拼在一起就是成品 |
| 函数式 | 榨汁机：苹果进去 → 苹果汁出来，苹果本身不受伤 |
| 声明式 | 去餐厅点菜："来碗牛肉面"，不用交代先烧水还是先切肉 |

#### 什么时候用哪种

| 范式 | 适用场景 | 特征 |
|------|----------|------|
| 过程式 | 一次性脚本、简单工具 | 几行写完、跑完就扔、不需要复用 |
| OOP | 有明确"对象"的系统 | 数据和行为绑定、长期维护、多人协作 |
| 函数式 | 数据处理流水线 | 数据经过多道转换、不存中间状态 |
| 声明式 | 查数据、配规则 | 配置驱动：SQL / Dockerfile / k8s yaml / HTML |

实际写代码很少只用一种。Python 里可以同时用 class（OOP）+ `map(filter(...))`（函数式）+ 查 SQL（声明式）。

### 静态类型 vs 动态类型

| | 静态类型 | 动态类型 |
|------|----------|----------|
| 类型检查 | 编译时检查 | 运行时检查 |
| 变量声明 | 必须声明类型 | 不需要 |
| 优点 | 早发现错误，IDE 支持好 | 写起来快，灵活 |
| 缺点 | 写起来啰嗦 | 类型错误要运行时才发现 |
| 代表 | Java, C++, TypeScript | Python, JavaScript, Ruby |
