## 一、可视化基本原则

### Tufte 三大原则

| 原则 | 含义 | 实践 |
|------|------|------|
| **数据墨水比（Data-Ink Ratio）** | 用来展示数据的"墨水"占总"墨水"的比例最大化 | 去掉冗余的网格线、背景色、3D 效果、多余图例 |
| **图表垃圾（Chart Junk）** | 去掉不传递信息的装饰元素 | 不要用 3D 饼图、不要加花哨的背景图 |
| **数据密度（Data Density）** | 单位面积内展示的数据量 | 同等空间展示更多信息 |

**面试话术**："好的可视化遵循 Tufte 原则——最大化数据墨水比，去掉一切不传递信息的装饰。能不用 3D 就不用 3D，能不用饼图就不用饼图，能用折线不用柱状。"

### 视觉编码（Visual Encoding）

不同视觉通道的**精度排行**（从高到低）：

| 视觉通道 | 适合的数据类型 | 精度 |
|---------|--------------|------|
| 位置（坐标轴上的位置） | 定量数据 | **最高** |
| 长度 | 定量数据 | 高 |
| 角度/斜率 | 定量数据 | 中 |
| 面积 | 定量数据 | 中低 |
| 颜色饱和度/亮度 | 定量数据 | 低 |
| 颜色色相 | 分类数据 | — |
| 形状 | 分类数据 | — |

**关键结论**：比较数值用位置（柱状图/散点图），不要用角度（饼图）和面积。人的眼睛对位置差异最敏感，对角度和面积差异迟钝。

## 二、图表选择指南

这是面试中**最高频的问题**："给你一组数据，你选什么图表？"

### 快速决策流程

```
数据类型 → 你要表达什么？

分类数据（各省的销售额）
├── 比较大小 → 柱状图（横向）或条形图（纵向）
├── 看占比 → 饼图（≤5 类）或堆叠柱状图（>5 类）
└── 看排名 → 柱状图 + 排序

时间序列数据（每天的销售额）
├── 看趋势 → 折线图
├── 看周期性 → 多周期折线图
├── 看累计 → 面积图
└── 对比多个时间段 → 多系列折线图

连续数据（用户年龄 vs 消费金额）
├── 看相关性 → 散点图
├── 看分布 → 直方图
├── 看分组差异 → 箱线图
└── 看密度 → 热力图 / 等高线图

地理数据（各城市的销售额）
├── 精确位置 → 散点地图
├── 区域统计 → 区域填充地图（Choropleth）
└── 流量/路径 → 流向地图

关系/层级数据
├── 网络关系 → 力导向图
├── 层级关系 → 树图 / 旭日图
└── 流程 → 桑基图
```

### 各图表对比表

| 图表 | 用途 | 适合数据类型 | 常见误区 |
|------|------|------------|---------|
| **折线图** | 展示趋势 | 时间序列 | 类别太多（>5 条线）时很乱 |
| **柱状图** | 比较大小 | 分类数据 | 从 0 开始，否则误导 |
| **条形图** | 比较大小（多类别） | 分类数据，类别名长 | — |
| **饼图** | 看占比 | 分类数据（≤5 类） | **不要用 3D 饼图**；类别太多用堆叠柱状图 |
| **散点图** | 发现相关性 | 两个连续变量 | 点太多会重叠，需透明或抽样 |
| **热力图** | 看密度/模式 | 两个离散或分箱后的变量 | 颜色选择影响判断 |
| **箱线图** | 看分布/离群值 | 分类+连续 | 比均值+标准差提供更多信息 |
| **面积图** | 趋势+累计 | 时间序列（≤3 个系列） | 系列太多会遮挡 |
| **雷达图** | 多维对比 | 多维度数据 | 维度太多（>10）可读性差 |
| **漏斗图** | 转化流程 | 递减的阶段数据 | — |

**面试话术**："选图表先看数据结构和业务目标——是看趋势、比较、分布还是关系？趋势用折线，比较用柱状，分布用直方图或箱线图，关系用散点。特别注意柱状图必须从零开始，饼图类别不要超过 5 个。"

## 四、大屏可视化项目流程

```
数据采集层 ──→ 数据存储层 ──→ 数据处理层 ──→ API 服务层 ──→ 前端展示层
  │              │              │              │              │
业务系统日志    HDFS/Hive     Spark ETL      Spring Boot    Vue.js
  MySQL         Kafka         数据聚合         Node.js       ECharts
  API 对接      HBase         指标计算        WebSocket     DataV
```

### 完整流程

```
1. 数据源
   ├── 业务数据库（MySQL）→ Sqoop/DataX 抽取到 Hive
   ├── 日志文件 → Flume → Kafka → HDFS
   └── 第三方 API → 定时拉取

2. 数据仓库（Hive）
   ├── ODS 层：原始数据
   ├── DWD 层：清洗、标准化
   ├── DWS 层：轻度汇总（日/周/月汇总表）
   └── ADS 层：面向大屏的指标表

3. ETL 处理（Spark/Hive）
   ├── 昨天新增用户数、订单数、GMV
   ├── 近 7 日趋势、环比、同比
   └── 按城市/品类等维度拆分

4. API 服务层
   ├── 查询 HBase/MySQL/Redis 中的聚合结果
   ├── 提供 RESTful API（JSON 格式）
   └── WebSocket 实时推送增量数据

5. 前端渲染
   ├── 定时轮询 API（每 5 秒 / 每 30 秒）
   ├── ECharts 渲染图表
   └── 数字翻牌器、滚动表格等动效
```

### 大屏技术选型

| 方案 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **ECharts + Vue.js** | 通用大屏 | 生态好、组件多、易上手 | 需要前端开发 |
| **DataV**（阿里云） | 快速搭建 | 拖拽式、模板多 | 付费、定制性受限 |
| **FineReport** | 企业报表 | 支持复杂报表 | 交互性一般 |
| **Grafana** | 运维监控 | 时序数据强、告警 | 不适合复杂业务展示 |

**面试话术**："大屏项目的数据链路一般是这样：数据从业务库经 DataX/Sqoop 入 Hive，Spark 做 ETL 按维度和时间粒度预聚合，聚合结果存 MySQL 或 Redis，后端提供 API，前端用 ECharts 渲染。实时部分用 Kafka + WebSocket 推增量数据。"

## 五、Tableau / FineBI 等 BI 工具概览

### 主流 BI 工具对比

| 工具 | 定位 | 优点 | 缺点 | 适合 |
|------|------|------|------|------|
| **Tableau** | 可视化分析 | 拖拽灵活，图表美观，社区大 | 贵（$70/用户/月），数据处理弱 | 分析师个人探索 |
| **Power BI** | 微软生态 | 与 Excel/Office 集成好，便宜 | 数据清洗能力一般 | 微软技术栈企业 |
| **FineBI** | 国内企业 BI | 国产化、权限管理强、与帆软报表集成 | 灵活度不如 Tableau | 国内中大型企业 |
| **Metabase** | 开源轻量 | 免费、部署简单、支持 SQL | 功能较简单 | 初创公司/小团队 |
| **Superset** | 开源 | 功能丰富、支持大数据源 | UI 相对粗糙 | 有 Hadoop 栈的团队 |

### Tableau 面试点

```text
1. 维度（Dimension）vs 度量（Measure）
   - 维度：分类变量（品类、地区、日期）→ 蓝色胶囊
   - 度量：数值变量（金额、数量）→ 绿色胶囊

2. 连续 vs 离散
   - 连续（绿色）：形成轴
   - 离散（蓝色）：形成标题

3. LOD 表达式（Level of Detail）
   - FIXED: 忽略视图维度，按指定维度计算
   - INCLUDE: 在视图维度基础上额外加入维度
   - EXCLUDE: 在视图维度基础上去掉某些维度

-- 示例：FIXED 计算每个用户的累计消费（不受视图筛选影响）
{FIXED [user_id]: SUM([amount])}
```

## 六、Python 可视化选型指南

### 三大库对比

| 库 | 定位 | 优点 | 缺点 | 适合 |
|----|------|------|------|------|
| **matplotlib** | 底层绘图库 | 精细控制、与 matplotlib 生态兼容 | 代码量大、默认丑 | 学术论文、出版级图表 |

### 实战代码示例

```python
import matplotlib.pyplot as plt
import seaborn as sns
import plotly.express as px
import pandas as pd
import numpy as np

# 模拟数据
np.random.seed(42)
df = pd.DataFrame({
    "date": pd.date_range("2026-01-01", periods=365, freq="D"),
    "gmv": np.random.normal(10000, 2000, 365).cumsum() / 100,
    "orders": np.random.poisson(200, 365),
    "category": np.random.choice(["数码", "服装", "食品", "美妆", "家居"], 365),
    "city": np.random.choice(["北京", "上海", "广州", "深圳", "杭州"], 365),
})

# ===== matplotlib：精细控制 =====
fig, axes = plt.subplots(2, 2, figsize=(12, 10))

# 趋势图
axes[0, 0].plot(df["date"], df["gmv"], color="#5470C6", linewidth=1.5)
axes[0, 0].set_title("每日 GMV 趋势", fontsize=14, fontweight="bold")
axes[0, 0].set_xlabel("日期")
axes[0, 0].set_ylabel("GMV")
axes[0, 0].grid(True, alpha=0.3)

# 柱状图（各品类订单量均值）
cat_order = df.groupby("category")["orders"].mean().sort_values()
axes[0, 1].barh(cat_order.index, cat_order.values, color="#91CC75")
axes[0, 1].set_title("各品类日均订单量", fontsize=14, fontweight="bold")

# 直方图
axes[1, 0].hist(df["orders"], bins=30, color="#FAC858", edgecolor="white")
axes[1, 0].set_title("订单量分布", fontsize=14, fontweight="bold")
axes[1, 0].axvline(df["orders"].mean(), color="red", linestyle="--",
                   label=f'均值={df["orders"].mean():.0f}')
axes[1, 0].legend()

# 饼图
cat_counts = df["category"].value_counts()
axes[1, 1].pie(cat_counts.values, labels=cat_counts.index,
               autopct="%1.1f%%", colors=sns.color_palette("pastel"))
axes[1, 1].set_title("品类分布", fontsize=14, fontweight="bold")

plt.tight_layout()
plt.savefig("dashboard.png", dpi=150, bbox_inches="tight")
plt.show()

# ===== seaborn：快速出统计图 =====
# 箱线图：按品类看订单量分布
plt.figure(figsize=(10, 6))
sns.boxplot(x="category", y="orders", data=df, palette="Set2")
plt.title("各品类订单量箱线图")
plt.show()

# 热力图：城市×品类的 GMV 交叉表
pivot = df.pivot_table(values="gmv", index="category", columns="city", aggfunc="sum")
plt.figure(figsize=(10, 6))
sns.heatmap(pivot, annot=True, fmt=".0f", cmap="YlOrRd", linewidths=0.5)
plt.title("城市×品类 GMV 热力图")
plt.show()

# 散点图 + 回归线
plt.figure(figsize=(8, 6))
sns.regplot(x=df.index, y=df["gmv"], scatter_kws={"alpha": 0.3}, line_kws={"color": "red"})
plt.title("GMV 趋势与回归线")
plt.show()

# ===== plotly：交互式图表 =====
# 交互式折线图
fig = px.line(df, x="date", y="gmv", title="每日 GMV 趋势",
              labels={"gmv": "GMV", "date": "日期"})
fig.update_layout(hovermode="x unified")
fig.write_html("gmv_trend.html")  # 导出为可交互的 HTML

# 交互式散点图（含气泡）
fig = px.scatter(df, x="gmv", y="orders", color="category",
                 size="orders", hover_data=["city"],
                 title="GMV vs 订单量（按品类着色）")
fig.show()

# 动态动画（按时间播放）
fig = px.bar(df, x="category", y="orders", color="category",
             animation_frame=df["date"].dt.month,  # 按月动画
             title="各品类订单量变化（按月）")
fig.show()
```

### D3.js 概览

D3.js 是底层可视化库，比 ECharts 灵活但代码量大。面试中提及即可，不需要手写。

```text
ECharts vs D3.js：
- ECharts：封装度高，配置驱动，20 行出一个图表。适合业务开发。
- D3.js：底层操作 SVG/DOM，数据驱动，200 行出一个定制图表。适合创新可视化。
- 面试话术："日常用 ECharts，灵活度高选 D3.js。ECharts 做不了的特殊交互图表才会考虑 D3。"
```

## 八、面试高频考点

### 考点 1：图表选择

**问题**："给你一组销售数据，你选什么图表？"

**回答**：先看数据结构和业务问题。想看各地销售对比→柱状图；想看半年趋势→折线图；想看品类占比→饼图（≤5 类）或堆叠柱状图；想看 GMV 和订单量关系→散点图。核心原则：比较用位置（柱状图），趋势用线（折线图），关系用点（散点图），分布用直方图/箱线图。

### 考点 2：大屏数据方案

**问题**："让你设计一个实时大屏的数据链路，怎么做？"

**回答**：数据源→Kafka（缓冲）→Spark Streaming/Flink（实时聚合）→Redis（热数据）+ MySQL（历史数据）→Spring Boot API→前端 ECharts 渲染。实时部分用 WebSocket 推送增量，历史趋势部分用 HTTP 轮询。核心优化点：预聚合、缓存、数据降采样。

### 考点 3：可视化性能优化

**问题**："大屏数据量很大时怎么优化？"

**回答**：三个层面：1) 数据层：预聚合（不要在前端算）、降采样（对折线图只保留 200 个点）、分页/分区间加载。2) 渲染层：ECharts 用 Canvas 不要用 SVG，开启 large 模式，渐进渲染。3) 架构层：用 Redis 缓存聚合结果，别每次查原始表。

### 考点 4：大屏适配

**问题**："大屏怎么适配不同分辨率的屏幕？"

**回答**：用 CSS transform: scale 做等比缩放。监听 window resize 事件，根据设计稿尺寸（如 1920x1080）和实际屏幕尺寸计算缩放比例，对容器做 scale 变换。所有尺寸用相对单位（vw/vh/%/rem），避免用固定 px。

### 考点 5：数据墨水比

**问题**："怎么看一个可视化做得好不好？"

**回答**：看数据墨水比——展示数据的元素占比是否最大化。好的可视化：1) 没有冗余装饰（无 3D、无背景图），2) 坐标轴合理（柱状图从 0 开始），3) 颜色有意义（不是随便着色），4) 一眼就能看出数据的故事，不需要读十几秒。
