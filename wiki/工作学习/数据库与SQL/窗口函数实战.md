# 窗口函数实战

数据分析面试 SQL 部分，窗口函数考得最多。普通 GROUP BY 只能保留聚合结果，窗口函数**保留每一行，同时在窗口内计算聚合值**。

## 数据准备

假设有一张电商订单表 `orders`：

```sql
CREATE TABLE orders (
    order_id    INT PRIMARY KEY,
    user_id     INT,
    order_date  DATE,
    category    VARCHAR(20),
    amount      DECIMAL(10,2)
);

INSERT INTO orders VALUES
(1, 101, '2026-06-01', '数码', 299.00),
(2, 101, '2026-06-05', '数码', 199.00),
(3, 101, '2026-06-10', '服装', 159.00),
(4, 102, '2026-06-02', '食品', 89.00),
(5, 102, '2026-06-03', '食品', 128.00),
(6, 103, '2026-06-01', '数码', 899.00),
(7, 103, '2026-06-08', '家居', 399.00),
(8, 104, '2026-06-04', '美妆', 199.00),
(9, 104, '2026-06-04', '服装', 299.00),
(10, 104, '2026-06-12', '数码', 599.00);
```

## 一、ROW_NUMBER / RANK / DENSE_RANK

面试必问：三者的区别。

```sql
-- 每个用户按金额排自己的订单
SELECT
    user_id,
    order_id,
    amount,
    ROW_NUMBER()  OVER (PARTITION BY user_id ORDER BY amount DESC) AS rn,
    RANK()        OVER (PARTITION BY user_id ORDER BY amount DESC) AS rk,
    DENSE_RANK()  OVER (PARTITION BY user_id ORDER BY amount DESC) AS dr
FROM orders
ORDER BY user_id, amount DESC;
```

| user_id | order_id | amount | rn | rk | dr |
|---------|----------|--------|----|----|-----|
| 101 | 1 | 299.00 | 1 | 1 | 1 |
| 101 | 2 | 199.00 | 2 | 2 | 2 |
| 101 | 3 | 159.00 | 3 | 3 | 3 |
| 102 | 5 | 128.00 | 1 | 1 | 1 |
| 102 | 4 | 89.00 | 2 | 2 | 2 |

**如果金额相同呢？**
| amount | ROW_NUMBER | RANK | DENSE_RANK |
|--------|-----------|------|------------|
| 199 | 1 | 1 | 1 |
| 199 | 2 | 1 | 1 |
| 89 | 3 | 3 | 2 |

- **ROW_NUMBER**：无脑 1234 排下来，并列也分先后
- **RANK**：并列同号，下一名跳号（1,1,3）
- **DENSE_RANK**：并列同号，下一名不跳号（1,1,2）

**面试话术**："ROW_NUMBER 是行号，RANK 是跳号排名，DENSE_RANK 是连续排名。需要取 Top N 每组且要处理并列时，用 RANK 或 DENSE_RANK。"

## 二、SUM / AVG 窗口聚合

```sql
-- 每个用户的累计消费金额（按时间排序）
SELECT
    user_id,
    order_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY user_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS cumulative_spend
FROM orders
ORDER BY user_id, order_date;
```

| user_id | order_date | amount | cumulative_spend |
|---------|-----------|--------|-----------------|
| 101 | 2026-06-01 | 299.00 | 299.00 |
| 101 | 2026-06-05 | 199.00 | 498.00 |
| 101 | 2026-06-10 | 159.00 | 657.00 |

**窗口帧（Window Frame）三个常用写法**：

```sql
-- 1. 累计（默认行为）
ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW

-- 2. 移动平均（最近3行）
ROWS BETWEEN 2 PRECEDING AND CURRENT ROW

-- 3. 整组（所有行）
ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
```

## 三、LAG / LEAD：前后行访问

```sql
-- 每个用户两笔订单之间的间隔天数
SELECT
    user_id,
    order_date,
    amount,
    LAG(order_date)  OVER (PARTITION BY user_id ORDER BY order_date) AS prev_order_date,
    LAG(amount)      OVER (PARTITION BY user_id ORDER BY order_date) AS prev_amount,
    DATEDIFF(order_date,
        LAG(order_date) OVER (PARTITION BY user_id ORDER BY order_date)
    ) AS days_since_last
FROM orders
ORDER BY user_id, order_date;
```

| user_id | order_date | amount | prev_order_date | prev_amount | days_since_last |
|---------|-----------|--------|-----------------|-------------|-----------------|
| 101 | 2026-06-01 | 299.00 | NULL | NULL | NULL |
| 101 | 2026-06-05 | 199.00 | 2026-06-01 | 299.00 | 4 |
| 101 | 2026-06-10 | 159.00 | 2026-06-05 | 199.00 | 5 |

**面试常见题**："找出金额比上一笔订单高的所有订单"：

```sql
SELECT order_id, user_id, amount
FROM (
    SELECT *,
        LAG(amount) OVER (PARTITION BY user_id ORDER BY order_date) AS prev_amount
    FROM orders
) t
WHERE amount > prev_amount;
```

### LAG/LEAD 进阶用法

**取前第 N 行 / 后第 N 行**：

```sql
-- LAG(column, offset, default)
SELECT
    user_id,
    order_date,
    amount,
    LAG(amount, 1, 0) OVER (PARTITION BY user_id ORDER BY order_date) AS prev_1_amount,
    LAG(amount, 2, 0) OVER (PARTITION BY user_id ORDER BY order_date) AS prev_2_amount,
    -- 取前第 2 笔订单金额，没有则填 0
    LEAD(amount, 1, 0) OVER (PARTITION BY user_id ORDER BY order_date) AS next_amount
    -- 取下一笔订单金额，没有则填 0
FROM orders
ORDER BY user_id, order_date;
```

| 参数 | 说明 | 示例 |
|------|------|------|
| `column` | 要取值的列 | `amount` |
| `offset` | 偏移行数，默认 1 | `2` 表示往前/后第 2 行 |
| `default` | 边界外无值时的默认值 | `0`、`'N/A'`、`NULL` |

**面试话术**："LAG 和 LEAD 的第二个参数控制偏移行数，第三个参数设置边界默认值。不设默认值时，第一行的 LAG 和最后一行的 LEAD 返回 NULL。"

## 四、FIRST_VALUE / LAST_VALUE / NTH_VALUE

这几个函数用来取窗口内的第一个值、最后一个值、第 N 个值。

```sql
SELECT
    user_id,
    order_date,
    amount,
    -- 每个用户的第一笔订单金额
    FIRST_VALUE(amount) OVER (
        PARTITION BY user_id ORDER BY order_date
    ) AS first_order_amount,
    -- 每个用户的最后一笔订单金额
    LAST_VALUE(amount) OVER (
        PARTITION BY user_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_order_amount,
    -- 每个用户的第 2 笔订单金额
    NTH_VALUE(amount, 2) OVER (
        PARTITION BY user_id ORDER BY order_date
    ) AS second_order_amount
FROM orders
ORDER BY user_id, order_date;
```

| user_id | amount | first_order_amount | last_order_amount | second_order_amount |
|---------|--------|-------------------|-------------------|---------------------|
| 101 | 299.00 | 299.00 | 159.00 | 199.00 |
| 101 | 199.00 | 299.00 | 159.00 | 199.00 |
| 101 | 159.00 | 299.00 | 159.00 | 199.00 |

**LAST_VALUE 的坑**：默认窗口帧是 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`，LAST_VALUE 在当前窗口帧内只会取到当前行，所以必须显式指定 `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING` 才能取到真正的最后一个值。

**面试话术**："FIRST_VALUE 和 LAST_VALUE 的默认窗口帧不一样。FIRST_VALUE 天然能取到第一个值（因为从第一行到当前行），LAST_VALUE 则必须改窗口帧到整组，否则取到的就是当前行本身。这是个常见坑。"

## 五、分布函数：PERCENT_RANK / CUME_DIST / NTILE

面试喜欢问 NTILE，因为它可以用来做用户分桶。

### NTILE：等分分桶

```sql
-- 把用户按消费金额分为 4 个层级
WITH user_total AS (
    SELECT user_id, SUM(amount) AS total_amount
    FROM orders
    GROUP BY user_id
)
SELECT
    user_id,
    total_amount,
    NTILE(4) OVER (ORDER BY total_amount DESC) AS tier
FROM user_total
ORDER BY total_amount DESC;
```

| user_id | total_amount | tier |
|---------|-------------|------|
| 103 | 1298.00 | 1 |
| 101 | 657.00 | 1 |
| 104 | 1097.00 | 2 |
| 102 | 217.00 | 4 |

**实战**：在用户运营中，NTILE(4) 把用户分成高/中高/中低/低四个层级，各层用户做不同的营销策略。

### PERCENT_RANK：百分比排名

```sql
-- 每个用户消费金额在自己分组中的百分位排名
SELECT
    user_id,
    SUM(amount) AS total_amount,
    ROUND(PERCENT_RANK() OVER (ORDER BY SUM(amount)), 2) AS pct_rank
FROM orders
GROUP BY user_id
ORDER BY total_amount;
-- pct_rank = (rank - 1) / (total_rows - 1)，范围 [0, 1]
```

### CUME_DIST：累积分布

```sql
-- <= 当前值的行数 / 总行数
SELECT
    user_id,
    SUM(amount) AS total_amount,
    ROUND(CUME_DIST() OVER (ORDER BY SUM(amount)), 2) AS cume_dist
FROM orders
GROUP BY user_id
ORDER BY total_amount;
-- 例如 cume_dist = 0.75，说明 75% 的人消费金额 ≤ 你
```

| 函数 | 公式 | 用途 |
|------|------|------|
| `PERCENT_RANK` | `(rank - 1) / (N - 1)` | 相对排名：你在前百分之多少 |
| `CUME_DIST` | `<= 当前值的行数 / N` | 累积占比：多少人 ≤ 你 |
| `NTILE(n)` | 等分 n 桶 | 用户分层、分位数 |

## 六、窗口帧（Window Frame）完整语法详解

窗口帧定义了窗口函数在每一行计算时"看哪些行"。这是理解窗口函数行为的关键。

### 完整语法

```sql
{ ROWS | RANGE | GROUPS }
BETWEEN frame_start AND frame_end
```

`frame_start` / `frame_end` 可选值：
- `UNBOUNDED PRECEDING` — 分区第一行
- `N PRECEDING` — 当前行往前 N 行
- `CURRENT ROW` — 当前行
- `N FOLLOWING` — 当前行往后 N 行
- `UNBOUNDED FOLLOWING` — 分区最后一行

### ROWS vs RANGE vs GROUPS 的区别

这是面试中拉开差距的知识点。

假设数据（按 order_date 排序）：
```
| row | user_id | order_date | amount |
|-----|---------|------------|--------|
| 1   | 101     | 06-01      | 100    |
| 2   | 101     | 06-01      | 200    |  ← 同一天！
| 3   | 101     | 06-05      | 150    |
| 4   | 101     | 06-10      | 300    |
```

```sql
-- ROWS：按物理行数算
-- 当前行 + 前 1 行，就是物理上的前 1 行
SUM(amount) OVER (ORDER BY order_date ROWS BETWEEN 1 PRECEDING AND CURRENT ROW)

-- RANGE：按 ORDER BY 列的值算（逻辑范围）
-- 当前行 + 前 1 天内的所有行
-- 注意：06-01 有两条记录，当在第二行时，前 1 天内的行 = 两行都在范围内
SUM(amount) OVER (ORDER BY order_date RANGE BETWEEN INTERVAL 1 DAY PRECEDING AND CURRENT ROW)

-- GROUPS：按 ORDER BY 列的相同值分组算
-- 当前行 + 前 1 组（所有和当前行 order_date 差一个不同值的行）
SUM(amount) OVER (ORDER BY order_date GROUPS BETWEEN 1 PRECEDING AND CURRENT ROW)
```

| 维度 | ROWS | RANGE | GROUPS |
|------|------|-------|--------|
| 计算依据 | 物理行号 | ORDER BY 列的值 | ORDER BY 列的相同值组 |
| 同值行处理 | 逐行分别算 | 同值行结果相同（都看同一批行） | 同值行结果相同 |
| ORDER BY 要求 | 不需要（但通常有） | 必须有数值或日期列 | 必须有 ORDER BY |
| 性能 | 最快 | 较慢（需判断值范围） | 中等 |
| 典型场景 | 移动平均、累计 | 时间窗口聚合 | 分组前后比较 |

**关键例子：为什么 ROWS 和 RANGE 结果不同**：

```sql
-- 数据：06-01 有两笔订单 100 和 200
-- ROWS 2 PRECEDING（物理前两行）：
--   第 3 行（06-05, 150）：看第 1、2、3 行 → 100+200+150 = 450
--   第 4 行（06-10, 300）：看第 2、3、4 行 → 200+150+300 = 650

-- RANGE INTERVAL 2 DAY PRECEDING（前 2 天内）：
--   第 3 行（06-05, 150）：看 06-03 ~ 06-05 的行 → 只有第 3 行 → 150
--   第 4 行（06-10, 300）：看 06-08 ~ 06-10 的行 → 只有第 4 行 → 300
--   因为数据中间隔了 4-5 天，前 2 天没数据
```

**面试话术**："ROWS 是按行号数，最简单也最快，90% 的场景够用。RANGE 是按值范围，适合做时间窗口聚合如'前 7 天'。GROUPS 在 MySQL 8.0+ 支持，按 ORDER BY 列相同值分组计算。"

## 七、实战：移动平均

计算 3 日 / 7 日 / 30 日 GMV 移动平均。

```sql
-- 先按天汇总 GMV
WITH daily_gmv AS (
    SELECT
        order_date,
        SUM(amount) AS daily_gmv
    FROM orders
    GROUP BY order_date
)
SELECT
    order_date,
    daily_gmv,
    -- 3 日移动平均（ROWS 方式：物理前 2 行 + 当前行）
    ROUND(AVG(daily_gmv) OVER (
        ORDER BY order_date
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ), 2) AS ma_3d,
    -- 7 日移动平均
    ROUND(AVG(daily_gmv) OVER (
        ORDER BY order_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ), 2) AS ma_7d,
    -- 30 日移动平均
    ROUND(AVG(daily_gmv) OVER (
        ORDER BY order_date
        ROWS BETWEEN 29 PRECEDING AND CURRENT ROW
    ), 2) AS ma_30d
FROM daily_gmv
ORDER BY order_date;
```

**RANGE 写法**（语义更准确，但要求日期连续无缺失）：

```sql
SELECT
    order_date,
    daily_gmv,
    -- 前 6 天 + 当天 = 7 天移动平均，缺失日期自动不出现在窗口内
    ROUND(AVG(daily_gmv) OVER (
        ORDER BY order_date
        RANGE BETWEEN INTERVAL 6 DAY PRECEDING AND CURRENT ROW
    ), 2) AS ma_7d
FROM daily_gmv
ORDER BY order_date;
```

| 写法 | 优点 | 缺点 |
|------|------|------|
| `ROWS 6 PRECEDING` | 快，逻辑简单 | 如果中间缺天，会取到更早期的数据（不精确） |
| `RANGE INTERVAL 6 DAY` | 语义准确（只看过去 7 天有数据的行） | 需要日期连续；性能稍差 |

## 八、窗口函数与自连接实现对比（EXPLAIN 分析）

以"每个用户累计消费金额"为例，对比窗口函数和自连接。

### 窗口函数方案

```sql
EXPLAIN SELECT
    user_id,
    order_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY user_id
        ORDER BY order_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS cumulative_spend
FROM orders;
```

EXPLAIN 结果分析：
```
id | select_type | table  | type | key              | Extra
1  | SIMPLE      | orders | ALL  | NULL             | Using filesort
-- 一次全表扫描 + 一次排序（按 user_id, order_date）
-- 时间复杂度 O(N log N)
```

### 自连接方案

```sql
EXPLAIN SELECT
    a.user_id,
    a.order_date,
    a.amount,
    SUM(b.amount) AS cumulative_spend
FROM orders a
JOIN orders b
    ON a.user_id = b.user_id
    AND b.order_date <= a.order_date
GROUP BY a.user_id, a.order_date, a.amount;
```

EXPLAIN 结果分析：
```
id | select_type | table | type | key    | rows | Extra
1  | SIMPLE      | a     | ALL  | NULL   | 10   |
1  | SIMPLE      | b     | ALL  | NULL   | 10   | Using where; Using join buffer
-- 两次全表扫描 + JOIN，rows = 10 × 10 = 100 行中间结果
-- 数据量越大差距越大：N 行 → 窗口函数 N 行，自连接 N² 行
```

| 维度 | 窗口函数 | 自连接 |
|------|----------|--------|
| 扫描次数 | 1 次 | N 次（每行扫一次） |
| 时间复杂度 | O(N log N) | O(N²) |
| 代码可读性 | 一行搞定 | 需要子查询/自连接 |
| 内存使用 | 窗口缓冲区 | 临时表 |

**面试话术**："能用窗口函数就用窗口函数。自连接在自增主键 + 索引覆盖的简单场景下性能还可以，但数据一大差距就是数量级的。"

## 九、面试专题：SQL 执行顺序

窗口函数和 GROUP BY 的执行顺序是面试高频考点。

### SQL 完整执行顺序

```text
FROM        ← 1. 确定数据来源，JOIN 也在这步
ON          ← 2. JOIN 条件
WHERE       ← 3. 行级过滤（不能用聚合函数）
GROUP BY    ← 4. 分组
HAVING      ← 5. 组级过滤（可以用聚合函数）
WINDOW      ← 6. 窗口函数计算 ← 在这！
SELECT      ← 7. 选择最终输出的列
DISTINCT    ← 8. 去重
ORDER BY    ← 9. 排序
LIMIT       ← 10. 限制行数
```

**关键结论**：
1. **窗口函数在 WHERE 之后执行** → WHERE 里不能用窗口函数，必须用子查询
2. **窗口函数在 GROUP BY 之后执行** → 窗口函数可以看到分组后的结果
3. **窗口函数在 SELECT 之前执行** → SELECT 里定义的计算列，窗口函数看不到（除非在子查询里）

```sql
-- ❌ 错误：WHERE 里用窗口函数
SELECT * FROM orders
WHERE ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY amount DESC) = 1;

-- ✅ 正确：用子查询包一层
SELECT * FROM (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY amount DESC) AS rn
    FROM orders
) t WHERE rn = 1;
```

**面试话术**："SQL 的执行顺序是 FROM → WHERE → GROUP BY → HAVING → WINDOW → SELECT → ORDER BY → LIMIT。记住窗口函数在 WHERE 和 GROUP BY 之后执行，所以 WHERE 里不能用窗口函数，但可以在 HAVING 后通过子查询间接使用。"

## 十、实战：用户充值流水账（余额变化）

**场景**：用户充值/消费表，需要计算每次操作后的余额。

```sql
-- 流水表：正数为充值，负数为消费
CREATE TABLE user_transactions (
    txn_id INT PRIMARY KEY,
    user_id INT,
    txn_date DATETIME,
    amount DECIMAL(10,2)  -- 正数=充值，负数=消费
);

INSERT INTO user_transactions VALUES
(1, 101, '2026-06-01 10:00', 100.00),   -- 充值 100
(2, 101, '2026-06-02 14:00', -30.00),   -- 消费 30
(3, 101, '2026-06-03 09:00', 200.00),   -- 充值 200
(4, 101, '2026-06-04 18:00', -50.00),   -- 消费 50
(5, 102, '2026-06-01 11:00', 500.00);

-- 查询每次交易后的余额
SELECT
    user_id,
    txn_date,
    amount,
    SUM(amount) OVER (
        PARTITION BY user_id
        ORDER BY txn_date
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS balance
FROM user_transactions
ORDER BY user_id, txn_date;
```

输出：
```
| user_id | txn_date            | amount | balance |
|---------|---------------------|--------|---------|
| 101     | 2026-06-01 10:00   | 100.00 | 100.00  |
| 101     | 2026-06-02 14:00   | -30.00 | 70.00   |
| 101     | 2026-06-03 09:00   | 200.00 | 270.00  |
| 101     | 2026-06-04 18:00   | -50.00 | 220.00  |
| 102     | 2026-06-01 11:00   | 500.00 | 500.00  |
```

**面试话术**："流水账用 SUM() OVER (PARTITION BY user_id ORDER BY time) 一行搞定，不需要存储过程循环计算。这就是窗口函数比游标/循环优雅的地方。"

## 十一、面试高频题：取每组 Top N

```sql
-- 每个品类金额最高的 2 笔订单
WITH ranked AS (
    SELECT *,
        ROW_NUMBER() OVER (PARTITION BY category ORDER BY amount DESC) AS rn
    FROM orders
)
SELECT category, order_id, user_id, amount
FROM ranked
WHERE rn <= 2
ORDER BY category, amount DESC;
```

**面试话术**："用窗口函数 ROW_NUMBER 分区排序，外层 WHERE 筛选排名。不用子查询反复嵌套。"

## 窗口函数 vs GROUP BY

| | GROUP BY | 窗口函数 |
|------|------|------|
| 结果行数 | 分组数（聚合后变少） | 原表行数（不变） |
| 用途 | 汇总统计 | 每行附加聚合信息 |
| 典型场景 | 每天的总销售额 | 每笔订单占当天销售额的比例 |

|------|----------|-----------|
| ROW_NUMBER / RANK / DENSE_RANK 区别 | 极高 | 并列处理、跳号逻辑 |
| LAG / LEAD 用法 | 极高 | 前后行访问、偏移参数、默认值 |
| 执行顺序 | 很高 | WHERE 不能用窗口函数的原因 |
| 窗口帧 ROWS vs RANGE | 高 | 物理行 vs 逻辑范围的差异 |
| NTILE 分桶 | 中高 | 用户分层实战 |
| 累计 / 移动平均 | 高 | 窗口聚合替代自连接 |
| FIRST_VALUE / LAST_VALUE | 中 | LAST_VALUE 的窗口帧陷阱 |
| 窗口函数 vs GROUP BY | 高 | 保留原行 vs 聚合压缩 |
| 窗口函数 vs 自连接 | 中 | EXPLAIN 对比性能差异 |
| PERCENT_RANK / CUME_DIST | 低中 | 分布分析 |
