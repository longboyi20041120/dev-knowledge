# Redis 实战：三大场景与分布式锁

> 缓存穿透/击穿/雪崩 + 分布式锁 SETNX/看门狗/RedLock——面试最高频。 | **面试重要度：极高** | 预计阅读：15 分钟

## 视频资源

- YouTube: [Hussein Nasser: Redis Internals](https://www.youtube.com/watch?v=4O4JRx7RwSY) — Redis 内部原理深度讲解
- B站: [尚硅谷 Redis 教程](https://www.bilibili.com/video/BV1Rv41177Af/) — 中文 Redis 最全教程

Redis 在项目里最常落地的三类场景，都是围绕它"**内存快 + 可设过期 + 单命令原子**"的核心特性展开的。这篇讲透三大场景，重点拆分布式锁——Java 后端面试的高频考点。

## 一、三大实用场景

| 场景 | 用到的特性 | 关键逻辑 |
|------|-----------|---------|
| **验证码登录** | 可设数据有效期 | 验证码存 Redis 并设过期时间，自动失效 |
| **查询缓存** | 读快 | 查询结果存 Redis，二次请求直读缓存 |
| **分布式锁** | SETNX 互斥 | 相同 key 只有一个线程能设置成功 |

### 查询缓存的一致性坑

数据库数据更新时，**要先删除对应缓存**，下次查询再把新数据同步进缓存。顺序错了（先更库后删缓存、或不删）会读到脏数据。

## 二、分布式锁：从 SETNX 到 RedLock

### 为什么需要

集群环境下，传统的 JVM 锁（synchronized、Lock）**只能锁住单台服务器**的线程。要跨服务器互斥，就得用分布式锁。

**典型场景**：电商秒杀减库存——多台机器同时减同一个库存，必须保证互斥。

### 核心原理

基于 Redis 的 **SETNX**（SET if Not eXists）命令：相同 key，只有第一个线程能保存成功，其余失败——天然的互斥性。

### 六个经典问题与解法

| 问题 | 现象 | 解决方案 |
|------|------|---------|
| **死锁** | 拿锁后宕机，锁不释放 | 加过期时间 |
| **过期时间难设定** | 设短了业务没跑完锁就没了 | **看门狗（watchdog）续期机制** |
| **误删** | A 的锁被 B 删了 | 用 **UUID 标识锁归属**，删前校验是不是自己的 |
| **原子性** | "判断+删除"两步之间出问题 | **原子性命令**（Lua 脚本） |
| **重入性** | 同一线程再次拿锁 | **哈希结构**记录重入次数 |
| **阻塞等待** | 拿不到锁空轮询浪费 CPU | **Redis 发布订阅**实现优雅唤醒 |
| **主从架构锁丢失** | 主节点宕机，锁还没同步到从节点 | **RedLock** |

### RedLock 与它的争议

RedLock 解决主从锁丢失：向**多个独立 Redis 节点**申请锁，多数成功才算拿到。

但它有争议：
- 运维成本高（要维护多个独立节点）
- 性能损耗
- 时钟同步风险（依赖各节点时间一致）

**替代方案**：Zookeeper 分布式锁（基于临时顺序节点，一致性更强）。
