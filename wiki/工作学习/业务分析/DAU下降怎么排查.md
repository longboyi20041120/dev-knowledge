# DAU 下降怎么排查

> 数据分析面试必考的 case 题，排查框架五步法。 | **面试重要度：极高** | 预计阅读：15 分钟

## 排查框架（按顺序，不要跳）

### 第一步：确认数据是否真实

**先排除数据问题，再分析业务问题。**

- 埋点有没有改动？SDK 升级了？上报断了？
- 数据管道有没有延迟？ETL 任务跑失败了？
- 统计口径变了？比如昨天改了 DAU 的定义？
- 对比其他指标：MAU、新增、启动次数也暴跌 → 大概率数据问题

> 面试话术："首先我会确认数据是不是准确，排查埋点、ETL、统计口径有没有变化。"

### 第二步：定位下降的维度

数据没问题，就要**下钻看是谁掉了**。

| 维度 | 看什么 | 如果异常说明什么 |
|------|--------|-----------------|
| 地区 | 各省/市 DAU | 某个地区暴跌 → 可能是区域性事件（停电、自然灾害） |
| 渠道 | iOS vs Android | 单端暴跌 → 可能是版本问题、应用商店下架 |
| 版本 | 各版本活跃 | 新版本暴跌 → 新版本有 bug |
| 新老用户 | 新增 vs 老用户 | 新增暴跌 → 投放停了/渠道挂了；老用户暴跌 → 产品问题 |
| 时段 | 小时级 DAU | 某个时段暴跌 → 服务器宕机、推送没发 |

> 面试话术："我会按地区、渠道、版本、新老用户四个维度拆解，定位是哪个群体的 DAU 掉了。"

### 第三步：找外部原因

如果是整体下降，先看外面发生了什么：

- 竞品搞了大活动？（比如拼多多百亿补贴，淘宝 DAU 就掉）
- 节假日效应？（春节大家都走亲戚，工具 App 没人用）
- 政策法规？（游戏版号、金融监管）
- 负面舆情？（微博热搜、App Store 差评激增）

### 第四步：找内部原因

如果外部没问题，排查内部改动：

- 最近发了新版本？
- 改了推荐算法？
- 服务器有没有宕机？
- Push 推送是不是停了？
- 某个核心功能挂了？（比如抖音推荐流挂了）

### 第五步：交叉验证

用其他指标验证你的假设：

- 假设是"新版本 bug 导致老用户流失" → 看该版本崩溃率、卸载率是否升高
- 假设是"竞品活动抢了用户" → 看竞品 DAU 是否同步上涨
- 假设是"推送停了" → 看推送点击率数据

## 六维度拆解框架（细化版）

上面的五步是流程，下面展开每一步里具体"怎么看"的六个维度。

### 维度一：外部因素

不只是"看看新闻"，要有系统的排查清单。

**排查清单**：

| 检查项 | 怎么查 | 影响特征 |
|--------|--------|---------|
| 竞品动态 | 竞品 DAU/下载量同期是否暴涨；竞品是否有大版本更新或大促 | 整体 DAU 下降，竞品同期上涨 |
| 节假日 | 对比日历：今年 vs 去年同期的 DAU 走势 | 如果去年同期也降，是正常季节性波动 |
| 政策/监管 | 搜索行业关键词 + "新政/监管/整改" | 通常影响整个行业，非单一产品 |
| 负面舆情 | 微博热搜、应用商店差评数、社交媒体舆情监控 | 伴随卸载率上升 |
| 天气/灾害 | 按地区拆解 DAU，看是否是区域性异常 | 某个地区突然归零或暴跌 |

**SQL 示例 — 检查是否是季节性波动**：

```sql
-- 对比今年和去年同期的 DAU 走势
-- 如果是节假日影响，去年同期的曲线应该也下降
SELECT
    '2026' AS year,
    DATE(event_date) AS dt,
    COUNT(DISTINCT user_id) AS dau
FROM user_activity
WHERE event_date BETWEEN '2026-06-01' AND '2026-07-01'
GROUP BY DATE(event_date)

UNION ALL

SELECT
    '2025' AS year,
    DATE(event_date) AS dt,
    COUNT(DISTINCT user_id) AS dau
FROM user_activity
WHERE event_date BETWEEN '2025-06-01' AND '2025-07-01'
GROUP BY DATE(event_date)
ORDER BY dt, year;

-- 如果两年同期走势一致，基本确定是季节性因素
```

### 维度二：竞品影响

**SQL 示例 — 竞品 DAU 是否同步上涨**：

```sql
-- 前提：你有竞品的数据（第三方数据源如 QuestMobile、极光）
-- 如果没有，可以用 App Store 排名变化作为代理指标
SELECT
    date,
    our_product_dau,
    competitor_dau,
    ROUND((competitor_dau - LAG(competitor_dau, 7) OVER (ORDER BY date))
          * 100.0 / LAG(competitor_dau, 7) OVER (ORDER BY date), 2) AS competitor_wow_change
FROM market_data
WHERE date >= DATE_SUB(CURRENT_DATE, INTERVAL 30 DAY)
ORDER BY date DESC;

-- 如果我们的 DAU 下降当天，竞品 DAU 周环比暴涨 > 20%，很可能是竞品抢用户
```

### 维度三：版本问题

版本问题是内部排查的第一优先级，因为影响面大、容易验证。

**排查步骤**：
1. 看最近一次发版日期，是否与 DAU 下降时间吻合
2. 按版本号拆分 DAU，看哪个版本的 DAU 降了
3. 对比新老版本的崩溃率、启动成功率、功能使用率

**SQL 示例 — 按版本拆解 DAU**：

```sql
-- 按版本号拆分活跃用户数，找出有问题的版本
SELECT
    app_version,
    COUNT(DISTINCT user_id) AS dau,
    -- 与 7 天前同版本对比
    COUNT(DISTINCT user_id) - COUNT(DISTINCT CASE
        WHEN event_date = DATE_SUB(CURRENT_DATE, INTERVAL 7 DAY) THEN user_id
    END) AS dau_change_vs_7d_ago,
    -- 崩溃率
    COUNT(DISTINCT CASE WHEN event_type = 'crash' THEN user_id END) * 100.0
        / COUNT(DISTINCT user_id) AS crash_rate,
    -- 启动成功率
    COUNT(DISTINCT CASE WHEN event_type = 'app_launch_success' THEN user_id END) * 100.0
        / COUNT(DISTINCT CASE WHEN event_type = 'app_launch' THEN user_id END) AS launch_success_rate
FROM user_activity
WHERE event_date = CURRENT_DATE - INTERVAL 1 DAY
GROUP BY app_version
ORDER BY dau DESC;

-- 预期输出示例：
-- +-------------+-------+---------------------+-----------+-------------------+
-- | app_version | dau   | dau_change_vs_7d_ago| crash_rate| launch_success_rate|
-- +-------------+-------+---------------------+-----------+-------------------+
-- | 4.2.0       | 50000 |               -12000|      8.50%|             91.20%|  ← 新版本，crash_rate 高
-- | 4.1.0       | 30000 |                 +1000|      1.20%|             99.50%|
-- +-------------+-------+---------------------+-----------+-------------------+
-- 解读：4.2.0 的崩溃率 8.5% 远超 4.1.0 的 1.2%，且 DAU 下降了 12000。
-- 基本确认：新版本有严重 bug。
```

### 维度四：渠道变化

渠道问题通常影响"新增 DAU"，不会直接影响老用户 DAU。

**排查清单**：

| 检查项 | 怎么查 | 影响特征 |
|--------|--------|---------|
| 投放是否暂停 | 看广告后台的投放状态和消耗金额 | 新增用户骤降 |
| 渠道转化率是否下降 | 当天的曝光→点击→下载转化率 vs 上周 | 素材老化或落地页挂了 |
| 应用商店是否变化 | App Store / 应用宝等排名变化；是否被降权 | iOS 或 Android 单端下降 |
| CPA 是否异常 | 投放费用 / 新增用户数 | CPA 暴涨 → 流量质量下降 |

**SQL 示例 — 按渠道拆分新增用户**：

```sql
-- 按渠道拆分新增用户，看哪个渠道的新增暴跌
SELECT
    channel,
    DATE(register_time) AS dt,
    COUNT(DISTINCT user_id) AS new_users,
    COUNT(DISTINCT user_id) - LAG(COUNT(DISTINCT user_id), 1) OVER (
        PARTITION BY channel ORDER BY DATE(register_time)
    ) AS change_vs_prev_day
FROM user_info
WHERE register_time >= DATE_SUB(CURRENT_DATE, INTERVAL 7 DAY)
GROUP BY channel, DATE(register_time)
ORDER BY dt DESC, new_users DESC;

-- 如果某个渠道的新增从 5000 突然降到 50，检查该渠道的投放状态和落地页
```

### 维度五：运营活动

活动结束后的"回落效应"是 DAU 下降的常见原因。

**排查要点**：
- 之前有没有大促/签到/红包活动刚结束？
- 活动期间 DAU 虚高了多少？（活动期间的 DAU 中，有多少是纯活动驱动的？）
- 活动结束后 DAU 回落的幅度是否在预期范围内？

正常情况下，活动结束后 DAU 应该回到"活动前基线 + 活动期间转化的留存用户"。如果 DAU 比活动前更低，说明活动可能有反作用（用户体验被透支）。

### 维度六：技术故障

技术问题是"骤降型"DAU 下降的第一大原因——不是用户不来了，是用户来了但没被统计到（埋点挂了），或者根本进不来（服务器挂了）。

**排查清单**：

| 检查项 | 监控来源 | 影响特征 |
|--------|---------|---------|
| 服务端错误率 | API 监控（5xx 比例） | 错误率飙升，DAU 骤降 |
| 服务端响应时间 | API 监控（P99 延迟） | 响应慢 → 用户超时退出 → DAU 降 |
| CDN 可用性 | CDN 监控 | 前端资源加载失败 → 白屏 → DAU 降 |
| DNS 解析 | DNS 监控 | 部分地区 DNS 解析失败 → 区域性 DAU 归零 |
| 推送服务 | 推送后台的到达率/点击率 | 推送到达率归零 → 通过推送唤回的 DAU 丢失 |
| 登录服务 | 登录成功率 | 登录失败 → 用户进不来 |

## 下钻分析：拆解维度矩阵

确定了下降范围后，需要一个系统化的拆解矩阵帮你定位到最小可操作的群体。

### 一级拆解：大的方向

| 一级维度 | 正常结论 | 待继续拆解的结论 |
|----------|---------|-----------------|
| 按地区 | 全国均匀下降 | 降幅分布不均（如华南 -30%，华北 -5%）→ 进入二级 |
| 按端 | iOS 和 Android 同步下降 | 单端下降 → 进入该端的二级 |
| 按新老 | 新增和老用户同步下降 | 新增降、老用户稳 → 投放/渠道问题；老用户降、新增稳 → 产品问题 |
| 按时段 | 全天均匀下降 | 集中在某个时段下降 → 服务器/推送问题 |

### 二级拆解：定位具体问题

| 一级结论 | 二级维度 | 正常结论 | 待深入结论 |
|----------|---------|---------|-----------|
| iOS 单端下降 | 按 iOS 版本 | 各版本均匀 | 集中在某个 iOS 系统版本 → App 与该版本兼容性问题 |
| 老用户下降 | 按用户生命周期 | 各阶段均匀 | 集中在 7-30 天用户 → 产品体验没留住中期用户 |
| 华南地区下降 | 按城市 | 华南各城市均匀 | 集中在某城市 → 可能是本地事件（网络故障、天气） |
| 集中在 20:00-22:00 | 按功能模块 | 各模块均匀 | 集中在某功能（如直播）→ 该功能挂了 |

### 三级拆解：交叉分析

当一级和二级都不能给出确定结论时，需要交叉拆解。

例如：不是简单的"iOS 下降了"或"老用户下降了"，而是**"iOS 上的老用户下降了，但 Android 上的老用户没变"**——这说明不是产品整体的问题，而是 iOS 端特定版本或体验的问题。

**交叉拆解组合**：
- 地区 x 渠道：是不是某个地区某个渠道的投放停了？
- 版本 x 端：是不是新版本只在 iOS 上发了，Android 没发？
- 时段 x 地区：某个地区白天正常、晚上暴跌 → 可能是该地区 CDN 晚间切了？

## 真实案例模拟：某天 DAU 下降 15%

下面是一个完整的排查过程演示，帮助你在面试中有故事可讲。

### 背景

某内容 App，日活约 200 万。2026 年 6 月 28 日（周日），日常数据监控告警：DAU 从前一天的 205 万降到 174 万，下降 15%。

### 时间线排查过程

**09:15 — 收到告警，第一步：确认数据是不是真的**

检查 ETL 任务状态：全部正常。检查埋点服务：上报量确实下降了 15%，不是漏报。对比启动次数和请求量：也下降了 14%-16%，基本确认是真实下降。

结论：数据是真的，进入业务排查。

**09:30 — 第二步：看是不是整体下降**

拉出过去 30 天 DAU 趋势图：6 月 28 日是近期最低点，且下降是骤降（前一天 205 万，第二天直接 174 万），不是缓慢下滑。

对比去年同期（2025 年同一周日）：去年同期也是小幅下降但只有 5%，今年 15% 远超季节性波动范围。

结论：不是正常的季节性波动，有异常事件。

**10:00 — 第三步：拆解维度定位**

按端拆分：iOS DAU 从 82 万降到 55 万（-33%），Android DAU 从 123 万降到 119 万（-3%）。

问题出在 iOS。

按 iOS 版本拆分：iOS 16.x 用户正常，iOS 17.x 用户正常，iOS 18.0-18.2 用户暴跌 40%。

再查 App 版本：我们 6 月 26 日发了 v5.3.0 版本。iOS 18.0-18.2 用户中，升级到 v5.3.0 的 DAU 下降了 52%；没升级的（还在 v5.2.x）下降仅 5%。

**10:30 — 锁定根因**

根因非常明确：v5.3.0 在 iOS 18.0-18.2 系统上有严重兼容性问题。

查崩溃率：v5.3.0 + iOS 18.0-18.2 组合的崩溃率高达 23%（正常 < 1%）。大量用户打开 App 就闪退，自然不来了。

**11:00 — 交叉验证**

- 卸载率：v5.3.0 用户的卸载率比 v5.2.x 高了 3 倍。
- App Store 评分：过去 24 小时差评激增，关键词"闪退"、"打不开"、"新版不会"。
- 客服工单：iOS 用户投诉量增加了 5 倍，主要反馈"更新后打不开"。

三条线索互相印证，根因确认。

**11:30 — 解决方案**

1. 紧急：iOS 端回滚到 v5.2.5（之前的热修复版本），发布紧急修复版 v5.3.1
2. 安抚：Push 通知受影响用户更新到修复版本，附带补偿（3 天会员）
3. 长期：建立版本灰度机制——新版本先推 5% 用户，观察 24h 崩溃率和留存后再全量

**结果**：修复版发布 48 小时后，iOS DAU 恢复到 79 万（恢复 93%），总 DAU 恢复到 198 万。

## 面试话术 + 追问应对

### 主问题："DAU 下降 20%，你怎么排查？"

**完整回答模板**：

> "我分五步来排查。
>
> **第一步，确认数据真实性**。检查埋点、ETL、统计口径有没有变化。同时看联动指标（启动次数、请求量）是不是也降了。如果只是 DAU 一个指标降而其他没降，大概率是数据问题。
>
> **第二步，定位下降维度**。按四个维度下钻：地区、渠道（iOS/Android）、版本、新老用户。目标是找到一个最小的异常群体——比如'iOS 新版本的老用户'而不是'所有人'。
>
> **第三步，排查外部因素**。竞品是否有大动作？是否节假日？是否有负面舆情？如果是行业性事件，应该有其他数据源（竞品、行业报告）能交叉验证。
>
> **第四步，排查内部因素**。最近是否有新版本发布？推荐算法是否调整？服务器或推送是否有故障？运营活动是否刚结束？
>
> **第五步，交叉验证假设**。如果我怀疑是版本问题，就看那个版本的崩溃率和卸载率。如果我怀疑是竞品影响，就看竞品数据。用多种数据源互相印证，确认根因后再给解决方案。"

### 追问 1："如果所有维度都正常，每个拆解都看不出问题，怎么办？"

> "这种情况说明下降不是集中在某个维度，而是均匀的、全局的。我会从三个方向继续挖。
>
> **方向一：数据层面再确认**。让工程团队把原始上报日志拉出来，重新非实时计算一遍 DAU。有时候实时计算的逻辑和离线逻辑有 diff。SDK 也可能有轻微改动（比如会话超时时间从 30 分钟改成 5 分钟），导致统计口径变化但没人通知分析团队。
>
> **方向二：看非结构化数据**。App Store 评论区、微博、小红书有没有用户在抱怨什么？有时候技术没问题但体验有问题（比如推荐算法的内容质量下降），用户觉得'变无聊了'就走了。这种下降是均匀的，因为所有用户看到的都是同质化的差内容。
>
> **方向三：看微观行为指标**。DAU 的数量级没变，但平均使用时长、每次会话深度（浏览页面数/观看视频数）、核心功能使用率是不是在下降？如果 DAU 数量没变但深度在持续衰减，说明产品在'温水煮青蛙'式地失去用户——这是最危险的，因为 DAU 这个滞后指标还没反映出来。"

### 追问 2："如果是缓慢下降（比如连续 3 个月每周跌 2%），不是骤降，怎么排查？"

> "缓慢下降和骤降的排查逻辑完全不同。骤降找事件（版本/故障/竞品），缓降找趋势（产品/市场/用户结构）。
>
> **第一步，确认下降的构成**。DAU = 新增 DAU + 老用户 DAU。把新增 DAU 和老用户 DAU 分别画趋势线，看是哪一端在降。如果新增 DAU 在降 → 获客效率下降（渠道质量变差、竞争加剧导致获客成本上升）。如果老用户 DAU 在降 → 留存变差。
>
> **第二步，群组分析（Cohort Analysis）**。按注册月份把用户分成群组，看每个群组的留存曲线是否在恶化。比如 2026 年 1 月注册的用户 30 日留存 15%，2026 年 5 月注册的用户 30 日留存只有 8%——同样的产品，新来的用户留不住，说明获客渠道的质量在下降，或者产品核心价值在减弱。
>
> **第三步，用户结构变化分析**。DAU 的下降可能是用户结构变化导致的。比如老用户（高活跃）占比下降，新用户（低活跃）占比上升。总 DAU 的下降实际上是'老用户自然流失 + 新用户活跃度不够'叠加的结果，而不是'用户变少了'。
>
> **第四步，对标行业和竞品**。如果整个行业的 DAU 都在降（比如工具类 App 整体下滑），那可能是用户需求在迁移。如果只有我们在降而竞品在涨，那就是产品竞争力的问题。"

### 追问 3："如果只影响 iOS 不影响 Android，怎么办？"

> "只影响一端恰恰是最容易排查的场景——排查范围直接缩小到 iOS 端特有的因素。
>
> **第一位：iOS 版本兼容性。** 检查最近是否发了新的 iOS App 版本。按 iOS 系统版本拆分——是某个系统版本的兼容性问题，还是所有 iOS 版本都受影响？如果是某个 iOS 系统版本的问题，让 QA 在这个版本上复现核心流程。
>
> **第二位：App Store 侧问题。** 检查 App 是否被 App Store 降权、下架、或因审核问题被移除某些地区。看 App Store Connect 后台的展示量和下载量是否异常。
>
> **第三位：iOS 特有的技术依赖。** 是否依赖了 iOS 独有的 API（如推送 APNs、Sign in with Apple、ATT 隐私弹窗）？相关服务是否正常？比如 ATT 弹窗审核未通过导致无法获取 IDFA，影响了广告归因，进而影响了广告投放的优化效果，间接导致获客下降。
>
> **第四位：iOS 用户画像差异。** iOS 用户和 Android 用户的画像通常不同（iOS 用户收入更高、一线城市占比更高）。如果只有 iOS 下降，可能反映的是高收入/一线城市用户群体的特定变化，而不是端的问题本身。用地区 x 端交叉拆解验证。"

## 面试高频考点

| 考点 | 考察什么 | 回答要点 |
|------|---------|---------|
| 排查框架的完整性 | 结构化思维 | 数据验证→维度拆解→外部→内部→交叉验证，顺序不能乱 |
| 先排查数据再排查业务 | 分析严谨性 | 面试官最看重的第一反应，不跳过数据验证直接分析业务 |
| 维度下钻能力 | 拆解思维 | 能说出 4+ 个拆解维度，知道每个维度异常说明什么问题 |
| 骤降 vs 缓降的不同排查逻辑 | 问题分类能力 | 骤降找事件（版本/故障/竞品），缓降找趋势（产品/市场/用户结构） |
| 交叉验证 | 假设验证能力 | 不满足于一个数据源，能用多个指标互相印证同一个假设 |
| 单端异常的排查 | 技术敏锐度 | 能想到系统版本兼容性、应用商店、端特有 API 依赖 |
| 真实案例讲述 | 实战经验 | 有完整的排查过程可以讲（时间线 + 数据 + 结论 + 结果） |
