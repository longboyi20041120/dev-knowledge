## 提问方法论

### 信息密度原则

**把 AI 不知道的说出来。** AI 不知道你的项目背景、技术栈、个人偏好、隐式约束。提问时提供上下文越充分，回答越精准。

对比：
- 差："这段代码报错了怎么办"
- 好："Spring Boot 3.x 项目，`PathServiceImpl.java` 第 45 行 `NullPointerException`，这段代码是 A* 导航算法的一部分"

### 中立提问

带主观情绪的提问会引导 AI 朝你预设的方向走，可能错过更好的方案：
- 差："这个垃圾框架是不是有问题"
- 好："这个框架在处理高并发场景下的表现如何，有没有替代方案"

### 多轮修正

频率越高，回答越准确。第一轮给方向，第二轮收窄，第三轮定稿。不要期望一轮完美。

### 写作流水线

```
先写大纲 → 修改大纲 → 拓充要点 → 生成全文
```

每一步都可以让 AI 先出草稿，你审核后再继续。AI 是大纲的最佳协作对象。

### 对比任务用 AI

综合对比（技术选型、工具评估、方案权衡）是 AI 的强项——能同时考虑多维度因素，不受个人偏好影响。

### 给 AI 标准

每份输出都应该给明确的约束：
- 格式（Markdown 表格 / JSON / 纯文本）
- 风格（正式 / 简洁 / 详细）
- 长度（200 字以内 / 不限）
- 禁止项（不要 Emoji、不要代码块）

## AI 的能力边界

| 能处理 | 说明 |
|--------|------|
| 文字 | 主要输入输出形式 |
| 代码 | 语法检查、重构、生成、审查 |
| 图片 | OCR 提取、图像分析（需 MCP 支持） |
| 语音 | 转文字后处理 |
| 结构化数据 | JSON、CSV、表格 |

| 不能 | 说明 |
|------|------|
| 训练截止后的知识 | 不知道 2025 年后的新事件、新框架版本 |
| 你脑子里的隐式约束 | 不说出来就不知道 |
| 实时数据 | 除非开 WebSearch 或 MCP 工具 |

## 信息获取渠道

| 渠道 | 适用场景 |
|------|----------|
| 原生预训练 | 通用知识、概念、语法 |
| WebSearch | 最新资讯、实时信息 |
| MCP 工具 | 系统文件、数据库、图片识别 |
