# Claude Code 进阶体系：Harness 与四大组件

多数人对 Claude Code 的认知只停留在"聊天写代码"，**据说实际利用率不足 10%**。真正拉开差距的不是模型本身，而是外面那层**框架（Harness）**——优化框架的收益，远超换一个更强的模型。这篇把一整套进阶体系串起来。

> 注：本篇整理自系列科普视频，部分数字（如"14 万 Token"）为视频的**粗略估算**，Claude 实际上下文为 200K，留了压缩和输出余量，理解方向即可，不必当精确值。

## 一、核心认知：决定上限的是 Harness，不是模型

Claude Code 的隐藏能力（完整记忆系统、任务拆分、安全检查）需要搭配 **Harness 框架**才能激活。Harness 分四层：

| 层 | 作用 | 关键点 |
|----|------|--------|
| 记忆层 | 解决模型"失忆" | 靠 **`CLAUDE.md`** 文件持久化项目规则（视频里误写成 `clone.md`，实为 CLAUDE.md） |
| 扩展层 | 能力扩展 | 支持多智能体协作、工具调用 |
| 集成层 | 流程自动化 | 把重复流程自动串起来 |
| 编码层 | 实际写代码 | 最终产出 |

**结论**：模型是发动机，Harness 是整台车。同一个模型，框架用得好不好，效果差好几倍。

## 二、五层记忆体系：解决"新会话失忆"

不用写代码，靠**分层的记忆文件**就能让 Claude Code 记住上下文：

| 层级 | 装什么 | 定位 |
|------|--------|------|
| 企业级 | 全局规范 | 整个组织统一标准 |
| 用户级 | 个人开发习惯 | 你自己的偏好 |
| 项目级 | 团队共享规范 | 一个项目的约定 |
| 规则级 | 模块化细粒度规则 | 针对具体模块 |
| 本地级 | 私密信息 | 不进版本库的敏感内容 |

从"组织 → 个人"层层细化，不同层管不同范围的记忆。

## 三、CLAUDE.md 写作范式

`CLAUDE.md` 是记忆层的核心文件，写好它有**三类核心内容 + 三大原则**：

**三类内容**：
1. 项目技术栈与框架
2. 项目规范
3. 执行命令

**三大原则**：
1. 告知项目目的（让 AI 知道这是干嘛的）
2. 划清 AI 权限边界（什么能动、什么不能碰）
3. 明确操作标准

关键是**精准**——内容太多会造成"上下文污染"，稀释关键信息。

## 四、四大组件：一句口诀记住

| 组件 | 一句话 | 作用层面 |
|------|--------|---------|
| **Commands 指令** | 斜杠命令封装高频重复操作 | 任务定义（如代码审查、提交代码，一键触发标准流程） |
| **Skills 技能** | MD 文档定义，语义感知**自动加载** | 执行路径（无需手动触发，需写清名称、职责、权限） |
| **Sub Agent 子智能体** | 拆分复杂任务，各有独立上下文 | 任务拆分（解决上下文窗口瓶颈） |
| **Hooks 钩子** | 特定事件触发，安全拦截 | 安全管控（检测并阻止危险指令） |

## 五、CLAUDE.md vs Skills：常驻 vs 按需

这是最容易混的一对：

| 维度 | CLAUDE.md | Skills |
|------|-----------|--------|
| 加载方式 | 常驻、全量加载 | 按需加载（语义触发） |
| 装什么 | 项目规则、架构等**通用知识** | 特定领域知识、专业**工作流** |
| 价值 | 避免模型失忆 | 节省上下文空间、避免关键信息被稀释 |

普通 prompt 式的 skills 写法，无法发挥 Claude 的主动工作能力——skills 的关键是**结构化封装**。

### Skills 的组成与合格标准

一个完整 skill 包含：
- **见名知意的名称**
- **SKILL.md**：定义核心任务与触发条件
- **scripts 脚本**：确定性逻辑用脚本做，省 Token
- **reference 参考文档**
- **模板**：指定输出格式

**两大合格标准**：① 精准的名称与触发描述；② 明确的工具权限范围。

## 六、子智能体（Sub Agent）：解决上下文污染

### 为什么需要

Claude Code 上下文窗口**可用容量约 14 万 Token（粗略）**，多任务并行时，冗余数据会挤占核心信息，导致"信噪比失衡"、代码写乱。子智能体的理论依据来自软件工程三大思想：

- **关注点分离**：一个智能体只干一类事
- **进程隔离**：各自独立上下文，互不污染
- **故障舱壁**：一个出问题不拖垮全局

### 怎么写

- 位置：`/claude/agents` 目录下的 Markdown 文件
- 结构：**YAML 前置元数据**（名称、描述、权限、运行配置）+ **Markdown 正文**（身份技能、工作流、输出格式）
- 铁律：**单一职责**，锁死权限（如"代码审查子智能体"只给读权限）

### 多子智能体流水线协作

单个子智能体不够，要**编排成流水线**。以 bug 处理为例：

```
bug 定位 → bug 修复 → bug 验证 → 报告撰写 → 汇总给主智能体
   ↓          ↓          ↓           ↓
 独立上下文   独立上下文   独立上下文    独立上下文
```

编排规则：职责不重叠、主智能体调度传递、上下游输入输出格式匹配、各自上下文隔离。

## 七、子智能体 × Skills 的两种协作模式

| 模式 | 逻辑 | 场景 |
|------|------|------|
| **预加载模式** | 子智能体启动时自动加载专业 skills，"开局自带技能包" | bug 修复前先加载修复规范类 skills |
| **派生调用模式** | skills 运行中遇到搞不定的任务，用 `contextfork` 创建子智能体协助 | 且子智能体上下文不污染主对话 |

差异核心：**系统提示词的控制权归谁**。

## 八、Hooks：唯一能"强制约束"的组件

前面几个组件（CLAUDE.md、skills、子智能体）都属于**认知层**——只是**建议性指导**。只有 **Hooks 属于系统执行层**，具备**强制拦截**能力。

- **拦截时机**：工具执行前 / 执行后 / 执行失败时
- **作用**：拦住大模型因随机性引发的危险操作（比如误删系统文件）
- **类比**：像 Java 的拦截器，到了对应环节自动触发预设逻辑

### 配置原则（决定故障率与信息捕获率）

- 早期基于 skills 的版本捕获率不足 80%，还会污染上下文
- 升级版用**请求前 + 请求后双拦截**，实现 100% 信息捕获 + 项目隔离
- **明确后缀词** 比语义感知的前置词更可靠
- **确定性逻辑优先用脚本**：降 Token 成本、避免误触发
