# Agentic AI 与 RAG：从"打字员"到"打工人"

一句话抓住本质：**Prompt 是"点"，Skill 是"线"，Agentic Workflow 是"完整的运行系统"**。这篇把 Prompt/Skill、Agent 工作流、Agentic Workflow、Agentic RAG 几个概念串起来，理清它们的关系。

## 一、Prompt vs Skill：点与线

| | Prompt | Skill |
|--|--------|-------|
| 解决什么 | **单点问题**（润色、翻译） | **一类问题**（论文写作、实验流程） |
| 形态 | 灵活直接的单次指令 | 多步骤标准化操作手册 |
| 优势 | 快速、精准 | 结构化封装，稳定可复用 |
| 适合 | 零散、临时需求 | 高频、重复的流程化任务 |

**选择逻辑**：
- 零散需求（术语翻译、语句润色）→ 用 **Prompt**
- 复杂任务（顶刊论文、批量数据处理）→ 用 **Skill**，一键调用整套流程

实操价值举例：一套"科研全流程 Prompt 大全"覆盖选题→综述→实验→投稿修回，用户只需替换可编辑区域，就能按规范输出。二者结合能显著提升 AI 协作效率。

## 二、Agent 工作流 → 可转化为自动化逻辑

**核心差异**：

- 普通自动化工作流：固定的"**如果…就…**"机械触发逻辑
- Agent 工作流：依赖**预设技能** + 调用**固定知识库**

**转化设想**：把 Agent 的技能、关联知识库及调用规则**提取整合**，就能形成适配特定场景的自动化工作流。已明确可适配的场景：**写短视频脚本、剪视频**等。

## 三、Agentic Workflow：让 AI 自己干活

普通 AI 需要人类**逐步骤下指令**；Agentic Workflow 只需人类**给出目标**，它自己拆解、执行、纠错、交付。

**核心循环**：

```
规划 → 执行 → 观察 → 反思 → （回到规划）
```

**组件关系**（用比喻理解）：
- **RAG** = AI 的"资料柜"（查资料）
- **ReAct** = AI 的"操作手"（动手做）
- 二者都是 Agentic Workflow 的**组成部分**，后者是完整的运行系统

**适用边界**：
- ✅ 适合：多步骤、复杂、重复的任务（能提效 10 倍、降错 80%）
- ❌ 不适合：简单单步、高精度要求、高创造性任务

**未来方向**：多智能体协作、长期记忆、自主学习。目前仍处于"1.0 版本"。

## 四、Agentic RAG：企业级的关键升级

传统 RAG 只能**被动检索**，Agentic RAG 引入智能体实现**主动推理 + 工具调用**，是工业级场景的关键方案。

### 传统 RAG 的三大局限

1. **检索相关性差**：关键词匹配易幻觉（把"服务器 502 报错"错配到"项目延期文档"，生成看似专业实则错误的回答）
2. **缺乏意图理解**：无法追问模糊问题的上下文（"报错"没说清系统/模块/时间）
3. **无法处理复杂任务**：不会拆解多步推理（"近三个月 A 产品退货率上升原因"需要 查数据→分析→给方案）

### Agentic RAG 的三大能力

- **查询重写**：把模糊问题转成专业描述（"报错怎么修"→"请提供服务器 502 报错日志"）
- **工具调用**：动态选向量库、内部 API、实时搜索等多源数据
- **反思迭代**：自动检查答案准确性，不足时重新检索或人工介入

### 对比一览

| 维度 | 传统 RAG | Agentic RAG |
|------|---------|-------------|
| 核心逻辑 | 检索 + 生成的被动流程 | 感知→决策→行动→反思的主动闭环 |
| 角色定位 | 资料朗读者 | 问题解决专家（像项目经理） |
| 典型场景 | 简单问答、关键词匹配 | 跨系统协作、复杂推理、企业级客服 |

一句话：Agentic RAG 靠"**能思考、能选择、能纠错**"，能解决企业 80% 以上的复杂需求。

## 附：Token 是什么

大模型的**计费单位**，类似电费里的"度"——用量直接挂钩费用。使用时不必纠结它到底是多少字节/字符，只需关注两点降本：**减少冗余内容传输、合理设置请求频率**。
